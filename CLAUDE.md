# CLAUDE.md

用于减少 LLM 常见编码错误的行为准则。可根据需要与项目专属指令合并使用。

**权衡说明:** 这些准则倾向于"谨慎优于速度"。对于琐碎任务,请自行判断是否套用。

## 1. 先思考,再编码

**不要假设。不要藏起困惑。把权衡摆到台面上。**

动手实现之前:

- 明确说出你的假设。如有不确定,就发问。
- 如果存在多种理解方式,把它们都列出来——不要沉默地替用户做选择。
- 如果有更简单的做法,直说。该反驳时就反驳。
- 如果有不清楚的地方,停下来。指明困惑在哪。然后发问。

## 2. 简单优先

**用解决问题所需的最少代码。不做任何投机性的工作。**

- 不实现用户没要求的功能。
- 不为只用一次的代码做抽象。
- 不增加用户没有要求的"灵活性"或"可配置性"。
- 不为不可能发生的场景写错误处理。
- 如果你写了 200 行,而 50 行就能搞定——重写。
- 用于用一个具体例子向你说明你的错误时，不要为了通过测试用例而硬编码一段只用于这个测试用例的代码。

问自己:"一个资深工程师会不会说这写得太复杂了?"如果会,就简化。

## 3. 外科手术式修改

**只动你必须动的地方。只清理你自己制造的烂摊子。**

编辑既有代码时:

- 不要"顺手优化"周边的代码、注释或格式。
- 不要重构没坏的东西。
- 配合既有风格,即使你自己更习惯另一种写法。
- 如果你注意到了无关的死代码,提一句——不要直接删掉。

当你的修改产生了孤儿代码时:

- 删除因你的修改而变得无用的 import、变量、函数。
- 不要删除原本就存在的死代码,除非用户要你删。

判断标准:每一行改动都应能直接追溯到用户的请求。

## 4. 目标驱动的执行

**先定义成功标准。然后循环执行直到通过验证。**

把任务转化为可验证的目标:

- "加上校验" → "为非法输入写测试,然后让它们通过"
- "修一下 bug" → "写一个能复现该 bug 的测试,包含若干个不同的测试用例，然后让它通过"
- "重构 X" → "确保重构前后测试都能通过"

对于多步骤任务,先简要陈述计划:

```
1. [步骤] → 验证:[检查项]
2. [步骤] → 验证:[检查项]
3. [步骤] → 验证:[检查项]
```

强成功标准让你能独立循环推进。弱标准("跑起来就行")会逼着你不断回头确认。

优雅的html文档是与用户讨论复杂任务的可靠模式。当用户为了与你更可靠的讨论而让你编写html时，参考 HTML_STYLE_GUIDE.md 中的内容。
该文件应该位于项目的根目录下，不过如果不在的话，请在项目中遍历子路径查找。如果该文件不存在于项目中，则提醒用户创建。
在与用户沟通的时候，除非用户明确要求，否则不要使用诸如“若把 Python 项目类比为一间厨房”这种方式和用户沟通。你的用户具有专业背景，这种类比没有任何实际意义。

## 5. git提交

**commit message应当能够明确说明你所做出的改动。**

每条 commit 由"动作标签 + 主段落 + (可选)子说明"三部分组成。
动作标签从下列里选一个,加冒号后接主段落:

- feat: 新增功能,或调整用户可见的行为
- fix: 修复 bug
- refactor: 改结构不改行为
- perf: 性能优化
- chore: 依赖、构建、CI、配置、清理死代码
- docs: 文档与注释
- test: 新增或修改测试

跨多类时,选最贴近读者关心点的那一类。选不出来,说明 commit 太大——拆开。
主段落描述"这次整体做了什么":

- 不少于 15 字。再短通常说不清"为什么"或"做了什么"。
- 不超过 150 字。再长通常意味着 commit 太大,或细节该下沉到子说明。
- 写成一段连贯的话,不要把要点直接塞进首行。

子说明用于强调值得被单独看见的具体改动:

- 每行以 - 开头,相对主段落缩进一级。
- 每行是单个句子,不超过 75 字——在 GitHub 网页上不应该换行。
- 只列读者可能错过的点:破坏性变更、隐式副作用、跨文件连带改动、关键数字。
- 没有值得强调的内容就不写,主段落已足够。

推送代码时，commit message不应该显示或隐式的刻意声明 AI 参与了工作

## 6. 安全

**项目的安全是你始终要考虑的问题。**

你在项目开发期间的所有行为，都应该遵循以下的安全指南：

- 任何账号，密码，API-key，都不应该被硬编码在代码文件中
- 上述敏感信息应该被写在配置文件中，代码运行时从配置文件读取
- 配置文件必须被写入 gitignore 文件中
- 如果你在阅读代码的过程中看到了硬编码的敏感凭据，第一实现向用户提出
- 你在工作过程中，只能从可信来源下载第三方库或拉取skill
