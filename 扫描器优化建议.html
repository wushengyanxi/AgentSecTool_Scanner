扫描器优化建议：从全量指纹库出发，下一步怎么把单 IP 研判做得更准

前提与结论

版本指纹库已覆盖全部 70 个正式版并经 30 个真实容器验证（详见《扫描研判依据与精准化路径》）。本篇基于这轮采集与验证中实测到的三个新事实——现实多数实例返回 401、只能靠指纹反推；6 组共 12 个版本资产指纹相同、只读手段无法区分；WS 只读握手拿不到版本号——给出分优先级、可落地、且每条都标明效果边界的优化建议。本篇严格区分“实测/源码证实”“待实测”“经核实不可行”三类，凡查证后站不住的方向也明确写出，避免误导工程投入。

01

先认清核心矛盾：为什么“指纹库全了”仍不够

指纹库覆盖到全量后，单 IP 研判的瓶颈从“数据缺失”转移到了三个结构性限制。这三个限制是本篇所有建议的出发点，每个都有实测依据。

限制
实测事实
对单 IP 研判的影响
多数实例无法直读版本 实测
control-ui-config 自 v2026.4.22 加鉴权（实测：4.12 返回 200、4.22 返回 401）；serverVersion 字段约自 v2026.3.12 才出现。
现实公网实例多为新版 → 返回 401 → 走不了直读，绝大多数目标只能靠资产指纹反推。指纹库的反推质量因此是研判精度的命门。
6 组版本不可区分 实测
纯后端改动的相邻版本前端 bundle 未变、资产指纹相同：1.12=1.13、1.23=1.24、3.7=3.8、4.7=4.8、4.20=4.21、5.5=5.6。
这 12 个版本若被 401 挡住版本号，反推只能到“两版之一”，探测器已如实返回 implicit-range 候选集。要再缩小，需要新的区分信号。
WS 只读拿不到版本 实测
实测对真实容器只读握手：服务端只主动推 connect.challenge 一帧，payload 仅 nonce/ts。带 server.version 的 hello-ok 帧需先发认证 connect 帧才回。
发认证帧违反只读原则，WS 这条路在合规约束下取不到版本。WS 信号的价值在判定（抗伪造），不在取版本。

三条限制共同决定：提升单 IP 研判精度的真问题，是“在只读约束下，为被 401 挡住、且落在不可区分组的实例，找到更多区分信号”，而非继续堆指纹数据。

02

P1：基于现有代码即可落地、收益直接

这一级都建立在已采集的证据或现有探测逻辑上，改动集中在 prober 侧，不依赖未实现的能力。

① 解析 connect.challenge 帧，把 WS 判定从“字符串匹配”升级为“结构化确认” P1 实测/源码证实

当前 ws.go:66 仅 strings.Contains 判断帧里是否含 "event":"connect.challenge"——任何服务只要响应里巧合带上这串就会被误判。实测确认该帧结构为 {"type":"event","event":"connect.challenge","payload":{"nonce":...,"ts":...}}。

改为 json.Unmarshal 并校验：type=="event" ∧ event=="connect.challenge" ∧ payload 含 nonce（UUID 形态）与 ts（数值）。

→ WS 信号从“可被字符串巧合伪造”升级为“结构化确认”，抗伪装，直接提升判定可信度。

边界：该帧不含版本信息（实测），此项只增强“是不是 OpenClaw”的判定，不解决取版本。不要期待它缩小不可区分区间。

② 把“未授权可访问”提升为一等研判输出 P1 源码证实

当 control-ui-config 返回 200（v2026.4.21 及更早、或新版未启用鉴权）时，配置元数据可被任意读取——这本身是高危事实。当前它只并入置信打分，未单独输出。

在 Result 增 unauthenticated bool：当 ControlUIStatus==200 且取到 serverVersion 时置真。

→ 让“这台配置可被任意读取”成为显式、可消费的风险结论，直接服务于风险分级与协同披露的优先级排序。

边界：返回 200 的实例现实占比低（多数已升级到 401），此标记命中的是少数老旧或错误配置的高危实例——恰恰是披露最该优先的对象。

③ 资产指纹反推加“部分匹配”兜底 P1 待实测确认

当前 MatchImplicit（fingerprints.go）要求资产名集合精确相等。现实中 CDN 改名、或某版只改了 js 没改 css，会导致集合不等而整体落空、退回无版本。

集合不完全相等时，按 js/css 命中比例给出候选版本 + 一个置信度，而非直接判空；返回结构复用已有的 version_candidates。

→ 缓解“部分升级 / CDN 改名”导致的整体丢失，提升反推召回。

边界：需在真实样本上验证“部分匹配”不会引入误配（两个无关版本恰好共享一个资产名）。落地前应先统计 71 版指纹库里 js/css 单独命中的碰撞率。

03

P2：需新增采集、作为增量佐证信号

这一级能提升判定鲁棒性或为未来留存证据，但效果需实测，且不直接解决不可区分问题。

④ 留存完整响应头与原始证据快照 P2 待实测确认

当前 Evidence（result.go）把三件套头压成一个 HeaderTriplet 布尔、CSP 只存哈希，Server 头、Content-Length/ETag/Last-Modified 等全部丢弃。

新增字段留存完整响应头快照与 control-ui-config 的 JSON 键集合；探测时一并落盘。

→ 既新增弱佐证信号，也为“未来发现某版本特有的头/键”留存原始材料——指纹维度可在资产名被 CDN 抹掉时多一条后路。

边界：源码核查未能定位到设置三件套头与 CSP 的服务端代码——但这与实测明确观测到这些头存在相矛盾，说明它们可能由中间件或未检索到的路径设置。落地前需先实测确认这些头的确切来源（OpenClaw 自身 vs 反代注入），否则可能把反代特征误当成 OpenClaw 指纹。

⑤ 采集 TLS 证书作为独立表面与去重键 P2 经核实 · 受限

设想是从 tls.ConnectionState 提取证书 CN/SAN/颁发者/自签状态/指纹，新增一个独立判定表面，并作为比 IP 更稳的资产去重键。

边界（重要）：源码确认 OpenClaw gateway 默认明文 HTTP（src/infra/tls/gateway.ts，TLS 默认 enabled=false）。现实里 TLS 多由前置反代提供，证书属于反代、不是 OpenClaw——用它判 OpenClaw 版本没有依据。

→ 仍有价值的部分：证书可用于资产聚类与去重（同一反代后的多实例）、以及判断“是否自部署”，但不能作为版本信号。建议只在去重/归属场景启用，不纳入版本判定。

04

经核实走不通的方向（避免无效投入）

以下方向看起来能缩小不可区分区间或补充版本信号，但查证后确认在只读约束或部署现实下站不住。明确列出，避免评估者重复投入。

设想方向
核实结论
为什么走不通
用 WS hello-ok 取 server.version 实测否定
实测：只读握手只收到 connect.challenge（仅 nonce/ts）。带 server.version 的 hello-ok 需先发认证 connect 帧。
发 connect 帧属认证/改状态动作，违反只读不变量（单测固化）。合规前提下这条路取不到版本。
主动请求区分 1.23/1.24、3.7/3.8 违反只读
源码：这 2 组确有协议级差异（stripMessageIdHints、systemInputProvenance 字段），但只在 chat send 请求/响应中体现。
触发这些差异需发 chat 消息——属改状态动作，越界。故这 2 组在只读约束下仍不可区分。
区分另外 4 组纯后端版本 原理不可行
源码：1.12=1.13、4.7=4.8、4.20=4.21、5.5=5.6 的改动全在后端逻辑/测试/出站请求头。
这些改动不产生任何外部可观测差异。任何只读手段都无法区分，应接受“报区间”为这些版本的最终结果。

结论：6 组不可区分版本里，没有任何一组能在只读约束下被进一步区分——这不是工程缺陷，是这些版本“前端无变化 / 差异仅在需鉴权的交互中”的固有性质。对它们，implicit-range 候选集就是诚实且最优的输出。

05

建议的落地顺序

按“收益/成本/合规风险”排序。P1 三项改动集中在 prober 侧、风险低、收益明确，建议优先；P2 两项需先实测前置事实再决定是否投入。

绿框＝建议优先做的低风险高收益项；橙框＝有条件的增益项，需先验证前置事实；深色框＝经核实在只读约束下走不通的方向，列出以避免无效投入。深色表示“终止区”，非“最重要”。

本篇延续《扫描研判依据与精准化路径》，把这轮指纹库全量化与 30 版验证中实测到的限制，转化为分级、可落地、标明边界的优化建议。每条建议的可信级别与边界均基于源码或真实容器实测，凡查证后站不住的方向（WS 取版本、主动请求区分、区分纯后端版本）一并写出，避免误导工程投入。下一阶段单 IP 研判精度的现实上限，由“被 401 挡住 + 落在不可区分组”的实例比例决定——P1 三项是当前性价比最高的推进方向。
