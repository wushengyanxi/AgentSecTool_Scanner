# Self-contained Project Docs HTML 生成提示文档

版本：v1.0  
用途：将一个工程项目的原始材料、已有说明文档、源码结构、运行命令、测试信息和维护约束，重构为一份可离线打开的 self-contained HTML 项目文档。  
目标形态：类似现代开发者文档站点，而不是博客、论文、展示页、PPT 或普通 README 润色版。

---

## 0. 使用方式

将本文件作为主提示词提供给 Claude、Codex 或其他代码智能体。随后附上项目材料，例如已有 HTML、README、目录树、源码片段、配置文件、schema、测试命令、运行输出、用户补充说明。模型的任务不是局部润色，而是基于全部输入材料重构一份完整的单文件 HTML 文档。

推荐输入顺序如下：

```text
[提示文档]
粘贴本文件全部内容

[项目材料]
1. 当前已有项目文档
2. 项目目录树
3. 核心源码文件或关键函数
4. 配置文件与环境变量
5. 数据库 schema / JSON / CSV / JSONL 样例
6. 构建、运行、测试命令
7. 运行输出或日志样例
8. 已知问题、未实现项、风险边界

[任务]
请根据以上材料生成一份完整 self-contained HTML 项目文档。
```

如果项目材料很大，应先让模型执行“文档审计与信息抽取”，确认缺失项、冲突项和待验证项，再生成最终 HTML。不要让模型在事实缺失时自行补全。

## 1. 任务定位

你不是在润色文章，也不是在美化 HTML。你是在把一个工程项目编译成一份可接手、可运行、可验证、可审计、可维护的单文件项目文档。

最终产物必须是一份 self-contained HTML 文档。它应具备现代开发者文档站点的信息结构：左侧导航、正文主栏、稳定锚点、quickstart、concepts、architecture、configuration、commands、reference、validation、troubleshooting、security boundaries、known limitations。文档应服务于工程接手与维护，而不是服务于展示、宣传、答辩或视觉包装。

目标读者包括人类维护者、代码审计者、运行操作者、项目评审者和代码智能体。读者应能在不询问原作者的情况下理解项目目标、跑通最小闭环、定位主要模块、解释关键规则、执行测试、判断风险边界、排查常见失败，并知道哪些内容不能随意修改。

---

## 2. 输出要求

最终输出必须是一个完整 HTML 文件源码。HTML 必须能直接保存为 `project-doc.html` 并在浏览器中离线打开。

HTML 必须满足以下约束：

- 使用 `<!DOCTYPE html>`、`<html lang="zh-CN">`、`<meta charset="UTF-8">`、响应式 viewport；
- CSS 全部内联在 `<style>` 中；
- 可选 JavaScript 必须内联在 `<script>` 中，且只能用于本地交互，例如复制代码、导航高亮、移动端菜单、表格过滤、折叠展开；
- 不得依赖外部 CSS、外部 JS、外部字体、外部图片、CDN、远程图标、远程地图或在线资源；
- 使用语义化结构：`header`、`nav`、`main`、`section`、`article`、`table`、`code`、`pre`；
- 每个一级章节必须有稳定 anchor；
- 页面必须包含左侧导航，移动端可折叠；
- 页面必须包含代码块、命令块、配置块、表格、callout、warning、status badge、reference card；
- 文档末尾必须包含 `Known Limitations / TODOs` 章节；
- 最终回答只输出 HTML 源码，不输出 Markdown，不输出解释性前言，不输出“下面是 HTML”。

如果输入信息不足以生成完整事实，仍然输出 HTML，但必须在相关章节写明 `UNKNOWN`、`TODO: source missing` 或 `UNVERIFIED`。不得编造不存在的命令、路径、字段、测试结果或验证结论。

## 3. 文档目标

合格的项目文档必须支持以下任务：

- 新接手者能在短时间内理解项目是什么、能做什么、不能做什么、当前做到什么程度；
- 新接手者能按 Quickstart 从干净环境跑通最小闭环；
- 维护者能定位核心模块、入口函数、配置项、数据对象和测试命令；
- 审计者能追踪关键结论来自哪些代码、测试、schema、日志或运行结果；
- 操作者能知道哪些命令会访问网络、写库、覆盖文件、消耗凭据或产生敏感数据；
- 代码智能体能获得稳定的项目上下文、修改边界、测试标准和 done 条件；
- 文档能够暴露未知项、未验证项、风险项，而不是用流畅叙述掩盖它们。

项目文档不是宣传页、论文讨论、开发日志、功能摘要、视觉展示页或单纯 README 扩写。若文档只能让读者“大致理解项目”，但不能支持运行、修改、验证和排错，它就不是完整项目文档。

## 4. 输入材料处理规则

你将收到的项目材料可能不完整、重复、过时或互相矛盾。处理优先级如下：

1. 实际源码、schema、测试文件、配置读取逻辑、运行输出优先；
2. 项目当前 README、已有文档、用户补充说明次之；
3. 注释和历史说明只能作为辅助线索；
4. 若材料冲突，必须在文档中显式标注冲突，不得悄悄选择一个看似合理的版本；
5. 若无法确定事实，使用 `UNKNOWN`、`TODO: source missing` 或 `UNVERIFIED`，不得脑补。

已有文档只能作为原始材料，不是必须保留的结构。你可以重排、拆分、合并、删除重复叙述、降低展示性元素权重、提升 Quickstart、Reference、Validation、Troubleshooting 的权重。不要因为原文是 HTML 就沿用其信息结构。不要因为原文某章很长就假定它完整。

## 5. 全局不可违反原则

### 5.1 文档是接手契约，不是说明散文

每个章节必须服务于明确的接手任务。自然段可以解释设计取舍，但不能替代模块契约、命令契约、数据契约、规则契约和验证契约。

### 5.2 不得只修局部

用户指出的问题只是症状。若用户指出某个术语未定义，你必须全文检查是否还有其他未定义术语。若用户指出某组规则只展开一条，你必须全文检查其他并列对象是否也存在非对称展开。不得只修改用户点名的一处。

### 5.3 不得用漂亮页面替代工程信息

视觉设计可以提升阅读体验，但不能替代命令、配置、状态、边界、验证和排错信息。展示性 hero、装饰图、颜色和卡片都不得降低信息密度。

### 5.4 不得把推断写成事实

任何“已实现”“已验证”“支持”“高置信”“安全”“只读”“幂等”“可恢复”“权威”“精确”等结论，都必须绑定代码、测试、命令、运行输出、schema、日志或人工复核方法。没有证据时必须标为待验证。

### 5.5 不得把辅助工具写成正式能力

项目中可能存在测试脚本、实验目录、数据供料模块、一次性迁移工具、外部平台拉取脚本、临时分析脚本。这些内容必须与正式能力区分。文档必须说明哪些模块位于正式主链路，哪些只是辅助链路、验证链路或实验链路。

## 6. 必须采用的信息架构

最终 HTML 必须包含以下一级章节。若输入材料缺失，仍保留章节并标注 `TODO` 或 `UNKNOWN`。

### 6.1 Overview

本节位于首屏，必须在最短路径内完成项目定位。必须包含：

- 项目一句话定义；
- 当前状态；
- 适用场景；
- 非目标；
- 核心能力；
- 核心约束；
- 权威数据位置；
- 最小运行路径入口；
- 继续阅读导航。

Overview 不应写成背景介绍。它必须让读者在短时间内知道项目是什么、能做什么、不能做什么、现在是否可运行、下一步该看哪里。

### 6.2 Quickstart

本节必须给出最小可复现闭环。要求从 clean checkout 或最小环境开始，说明必要依赖、配置准备、构建命令、最小输入、运行命令、预期输出、验证命令和清理方式。

若项目涉及公网、真实数据、凭据或安全敏感操作，Quickstart 必须使用本地 demo、mock、fixture、测试数据或受控目标，不得把生产扫描、真实 API 调用、真实用户数据处理作为最小路径。

每个 Quickstart 命令必须标注：

- 运行目录；
- 是否写文件；
- 是否写数据库；
- 是否访问网络；
- 是否需要凭据；
- 是否幂等；
- 失败后如何重跑；
- 预期输出或成功标志。

### 6.3 Architecture

本节说明系统结构。必须包含：

- 模块图或数据流图，使用纯 HTML/CSS/SVG 内联实现；
- 模块职责表；
- 每个模块的输入、输出、权威状态、依赖、失败行为；
- 正式主链路；
- 辅助链路；
- 实验性或测试性目录；
- 不参与正式能力的模块必须明确标注。

目录树可以出现，但只能作为索引，不能替代模块说明。路径旁边写一句注释不构成模块契约。

### 6.4 Core Concepts

本节定义项目中反复出现的核心概念。任何后文会使用的术语、缩写、编号、状态标签、规则名、数据对象、外部系统，都必须在这里或首次出现处定义。

每个概念必须包含：

- 名称；
- 定义；
- 在系统中的作用；
- 不应如何理解；
- 与相近概念的区别；
- 出现位置或实现位置。

不得出现“强证据”“弱证据”“三件套”“当前态”“快照”“只读”“候选”“权威”“高置信”等未落到具体对象的抽象词。

### 6.5 Workflows

本节描述项目的主要工作流。每条工作流必须写成从输入到输出的闭环，而不是功能摘要。至少应覆盖：

- 本地测试工作流；
- 正式运行工作流；
- 数据入库工作流；
- 结果查看工作流；
- 维护更新工作流；
- 若适用，报告、导出、披露或发布工作流。

每条工作流必须包含前置条件、步骤、命令、产物、验证方法、失败恢复、禁止事项。

### 6.6 Rules and Decision Logic

任何判定规则、分类规则、版本规则、去重规则、权限规则、写入规则、状态升级规则，都必须进入本节。规则不得只用自然语言解释，必须给出决策表、状态表或规则卡。

每条规则必须包含：

- 规则名称；
- 规则目标；
- 输入信号；
- 触发条件；
- 拒绝条件；
- 输出状态；
- 优先级；
- 与其他规则冲突时的处理；
- 信任假设；
- 误判来源；
- 漏判来源；
- 实现位置；
- 测试覆盖；
- 样例输入；
- 样例输出。

若列出一组并列规则，必须对称展开。不得只详细解释其中一条，而让其他规则停留在名称级别。若某规则暂时没有完整测试或数据支撑，必须显式标记。

### 6.7 Data Model

本节描述所有权威数据、派生数据、临时数据和外部数据。必须包含：

- 数据对象名称；
- 存储位置；
- 格式；
- 身份键；
- 去重语义；
- 覆盖语义；
- 追加语义；
- 时间字段语义；
- 空值语义；
- 是否权威；
- 是否可删除；
- 是否可提交到版本库；
- 是否包含敏感信息；
- 数据迁移或回填方式。

数据库表必须给出字段级说明。JSON、CSV、JSONL、配置文件也必须给出样例和字段解释。

### 6.8 Configuration

本节列出所有配置项、环境变量、凭据、默认值和配置文件。每一项必须包含：

- 名称；
- 类型；
- 默认值；
- 是否必需；
- 作用；
- 安全敏感性；
- 示例；
- 读取位置；
- 修改后是否需要重启；
- 错误配置时的表现。

不得只写“配置见 config/”。必须让读者知道每个配置项是否必须设置，以及错了会怎样。

### 6.9 Commands and Reference

本节是完整命令参考。每个命令必须包含：

- 命令；
- 用途；
- 运行目录；
- 前置条件；
- 输入；
- 输出；
- 副作用；
- 网络访问；
- 凭据需求；
- 幂等性；
- 失败恢复；
- 典型用例；
- 危险参数；
- 相关测试。

命令参考不得替代 Quickstart。Quickstart 是最小路径，Commands 是完整索引。

**命令样例（强制）**：命令参考中，每个命令或每组参数表之后，必须附带若干条可直接复制运行的命令样例，覆盖该命令最常见的参数组合。仅有参数表（flag/默认/作用）而无样例命令块的命令参考不合格。样例应包含：最小用法一条、加最常见参数的典型用法一到两条、以及对有危险副作用参数的用法（如有）。样例须放在可复制的 `<pre><code>` 代码块中。

### 6.10 Validation

本节说明如何验证项目结论。必须包含：

- 单元测试；
- 集成测试；
- 本地 fixture；
- 样例输入输出；
- 已验证版本或环境；
- 未覆盖项；
- 人工复核方法；
- 关键结论的证据来源。

凡正文中出现“已验证”“支持”“可用”“高置信”“安全”“只读”“幂等”“可恢复”等结论，必须能在本节找到对应验证方法。没有验证的结论必须改写为假设或待验证事项。

### 6.11 Failure Modes and Troubleshooting

本节必须按模块列出常见失败。每个失败项必须包含：

- 现象；
- 可能原因；
- 检查命令；
- 相关日志；
- 恢复方式；
- 是否会影响权威数据；
- 是否可以安全重跑；
- 何时需要人工介入。

不得只写“检查日志”。必须说明日志在哪里、看什么字段、什么输出代表问题成立。

### 6.12 Security, Privacy, and Operational Boundaries

若项目涉及网络、扫描、agent、自动执行、凭据、用户数据、安全测试、数据库写入、外部平台，本节必须详细。包含：

- 授权边界；
- 默认允许行为；
- 默认禁止行为；
- 网络请求类型；
- 速率限制；
- User-Agent 或身份标识；
- blocklist / allowlist；
- 凭据存储；
- 敏感数据脱敏；
- 日志留存；
- 误操作影响；
- 回滚和止损方式；
- 对外分享规则。

不得只写“只读”“安全”“授权范围内”。必须把这些词落到具体请求、具体状态修改、具体禁止行为和具体测试断言上。

### 6.13 Maintenance

本节说明项目如何维护。包含：

- 依赖升级；
- 规则库、指纹库、schema、配置或模型更新；
- 发布流程；
- 数据备份；
- 数据清理；
- 迁移流程；
- 新模块接入规范；
- 文档更新触发条件；
- 代码智能体修改时的规则。

### 6.14 Agent Instructions

本节专门面向代码智能体。必须包含：

- 项目目标压缩说明；
- 代码地图；
- 修改前必须阅读的文件；
- 禁止修改或需谨慎修改的路径；
- 常用测试命令；
- done 标准；
- 代码风格；
- 数据与凭据处理规则；
- 不得执行的命令或行为。

本节应接近 `AGENTS.md` / `CLAUDE.md` 的作用：让代码智能体在后续任务中继承项目约束。

### 6.15 Known Limitations and TODOs

本节集中列出所有缺失、未知、风险、待验证项。不得把 TODO 散落到正文后不汇总。每个 TODO 必须包含：

- 缺失内容；
- 影响范围；
- 为什么目前无法确定；
- 需要什么证据；
- 建议补齐位置。

### 7.5 配置项模板

```text
配置项：

是否必需：
来源：环境变量 / 配置文件 / 命令参数 / 编译常量
读取位置：
作用：
示例：
安全敏感性：
修改后是否需要重启：
错误配置表现：
```

## 8. 术语闭包审计

在生成最终 HTML 前，必须执行术语闭包审计。检查全文中是否存在以下问题：

- 首次出现但未定义的缩写；
- 名称像规则但没有规则定义；
- 名称像状态但没有状态含义；
- 名称像数据字段但没有字段语义；
- 名称像模块但没有输入输出；
- 名称像命令但没有副作用说明；
- 抽象词没有落到具体对象，例如“强证据”“弱证据”“当前态”“快照”“只读”“候选”“权威”“高置信”；
- 对同一概念使用多个名称；
- 同一名称在不同章节语义不同。

若发现问题，必须在正文中修复。无法修复时，加入 `Known Limitations / TODOs`。

---

## 9. 对称展开审计

当文档列出并列对象时，必须对称展开。并列对象包括但不限于：

- 模块列表；
- 规则列表；
- 测试项列表；
- 接口列表；
- 命令列表；
- 数据表列表；
- 配置项列表；
- 工作流列表；
- 状态列表；
- 风险列表。

对称展开不要求篇幅完全相等，但必须包含同一组核心字段。若 A 有定义、输入、输出、失败行为、实现位置、测试方法，那么 B 和 C 也必须至少有这些字段。不得只详细解释一个对象，其余对象留给读者猜测。

---

## 10. 证据绑定审计

以下词语出现时，必须检查是否有验证或证据：

- 已实现；
- 已验证；
- 支持；
- 覆盖；
- 高置信；
- 强证据；
- 弱证据；
- 安全；
- 只读；
- 幂等；
- 可恢复；
- 可复现；
- 不会；
- 保证；
- 权威；
- 当前态；
- 历史；
- 精确；
- 低风险。

若没有证据，必须改写为：

```text
当前设计意图是 ...
根据现有代码推断 ...
待验证：...
UNKNOWN: ...
TODO: 需要补充测试或运行输出
```

不得把推断写成事实。

---

## 11. 命令副作用审计

文档中的每个命令必须明确副作用。尤其检查：

- 是否访问公网；
- 是否访问内网；
- 是否需要 API key 或凭据；
- 是否写数据库；
- 是否覆盖文件；
- 是否追加日志；
- 是否删除数据；
- 是否启动长时间进程；
- 是否消耗外部平台配额；
- 是否可能触发安全敏感行为；
- 是否可以重复执行；
- 失败后能否安全重跑。

任何无法判断副作用的命令，都必须标注 `UNKNOWN side effect`，不得当作安全命令展示。

---

## 12. 数据权威性审计

必须区分以下数据类型：

- 权威数据：项目正式状态来源；
- 派生数据：可由权威数据重新生成；
- 临时数据：运行中间产物，可删除；
- 外部数据：来自第三方平台或外部系统，不能直接等同于项目确认事实；
- 测试数据：用于 fixture、demo 或验证，不代表生产事实；
- 敏感数据：含凭据、真实用户数据、真实目标、内部路径、日志原文等。

文档中每个数据对象都必须归入其中一类。不得把临时产物写成权威数据，不得把外部平台标注写成项目确认结论。

---

## 13. 安全边界审计

若项目涉及网络、自动执行、agent、扫描、安全测试、凭据、数据库写入或用户数据，必须执行安全边界审计。检查：

- 是否明确授权范围；
- 是否明确默认允许行为；
- 是否明确默认禁止行为；
- 是否说明实际发送的请求或执行的动作；
- 是否说明速率限制；
- 是否说明 allowlist / blocklist；
- 是否说明凭据存储和读取；
- 是否说明敏感数据脱敏；
- 是否说明日志留存；
- 是否说明误操作后果；
- 是否说明止损和恢复方式；
- 是否用测试或审计约束证明“只读”“安全”“不会破坏”。

不得只写原则性安全声明。安全声明必须映射到具体行为和具体限制。

---

## 14. HTML 交互要求

HTML 可以包含轻量本地交互，但必须无外部依赖。建议实现：

- 代码块复制按钮；
- 左侧导航当前章节高亮；
- 移动端导航折叠；
- 表格横向滚动；
- 可展开的长命令说明；
- 简单文档内过滤或搜索；
- 回到顶部按钮。

交互脚本必须简短、可读、无远程请求、无 `eval`、无外部库。若实现搜索会显著增加复杂度，可以省略，但导航和代码复制应优先保留。

---

## 15. CSS 设计要求

CSS 应采用开发者文档站点风格：

- 使用系统字体；
- 代码使用 monospace；
- 左侧导航固定，主栏滚动；
- 最大正文宽度适中；
- 表格清晰，长表格横向滚动；
- 代码块有复制按钮；
- 打印时隐藏导航，保留正文；
- 色彩克制，高对比度，避免视觉噪声。

**配色铁律（正文全灰度）**：正文一律使用灰度（黑 / 深灰 / 中灰 / 浅灰 / 灰底），**唯一允许的彩色是超链接的蓝色**（用于让链接可被识别为可点击）。具体：

- 不得使用彩色 callout。callout 一律灰底灰框，靠**标题文字**区分类型——警示用标题前缀 `WARNING`、`NOTE`、`INVARIANT`、`TODO` 等文字标明，**不得**用红 / 黄 / 绿 / 蓝色块或彩色边框表示警示或类型。原本会用红色表达的危险提示，改为灰底块 + 标题写 `WARNING`。
- 状态 badge（stable / partial / experimental / deprecated / unknown）一律**统一灰框灰底**，只靠 badge 内的文字区分状态，不得用绿 / 黄 / 紫等颜色编码状态。
- 图示（SVG / diagram）的节点、边框、连线一律灰度，不得用彩色色块；不得出现纯黑（或接近纯黑）的大面积填充色块——强调某节点应靠边框粗细、灰度深浅或文字，而非高饱和 / 纯黑底。
- 代码内的来源标记、证据标记等辅助文字一律中灰 / 浅灰，不得用紫 / 绿等彩色。

不得使用外部字体。不得使用远程图标。不得使用营销式渐变、大面积 hero、复杂动画或装饰性插图。文档的第一目标是检索、执行和审计。

---

## 16. 图示要求

若需要图示，优先使用内联 SVG 或纯 HTML/CSS diagram。图示必须服务于以下对象之一：

- 数据流；
- 模块边界；
- 运行路径；
- 状态转换；
- 规则决策；
- 权限边界；
- 数据权威性关系。

图示不能只是装饰。图示下方必须解释线型、颜色、节点、边界和方向含义。若图示无法准确表达系统事实，宁可使用表格。

**图示必须真实渲染核对（强制）**：任何 SVG / diagram 生成或修改后，**必须真正用浏览器打开该 HTML 并截图核对**渲染结果，不得仅凭源码推断。核对项至少包括：箭头是否对准目标节点（不偏移、不悬空）、连线是否压过文字或穿过其它节点、节点框是否互相重叠、文字是否溢出节点框、整体布局是否错位。发现问题必须修正后重新截图，直到渲染正确为止。这条同样适用于对任何含图示页面的视觉改动。

---

## 17. 状态标签规范

状态标签必须统一，禁止随意发明状态词。

```text
stable：正式路径或正式能力，已有基本验证。
partial：部分实现，能运行但覆盖不完整或存在已知缺口。
experimental：实验性、验证性、一次性或非正式能力。
deprecated：仍存在但不推荐继续使用。
unknown：输入材料不足，无法确定状态。
unverified：代码或文档声称存在，但缺少测试、运行输出或复核证据。
```

任何模块、命令、规则、数据对象若状态不明确，应标为 `unknown` 或 `unverified`，不得默认为 stable。

---

## 18. 对已有文档的重写要求

如果输入材料包含已有 HTML、README 或长文，你必须将其视为原始材料，而不是最终结构。必须特别处理以下常见问题：

- 只有解释，没有运行路径；
- 只有目录树，没有模块契约；
- 只有结论，没有证据；
- 只有正常流程，没有失败路径；
- 只解释某个规则，没有对称解释其他规则；
- 使用未定义术语；
- 把辅助模块写成正式能力；
- 把外部数据源写成项目确认事实；
- 命令缺少副作用说明；
- 数据对象缺少权威性说明；
- 安全边界只写口号；
- 文档视觉漂亮但不可接手。

你不能只修用户指出的一处。用户指出的问题只是一个症状，你必须全文寻找同类问题并统一修复。

---

## 19. 最终生成流程

你必须按以下流程处理，但最终只输出 HTML：

1. 读取全部输入材料；
2. 提取项目事实；
3. 区分事实、推断、未知、冲突；
4. 建立文档信息架构；
5. 执行术语闭包审计；
6. 执行对称展开审计；
7. 执行命令副作用审计；
8. 执行数据权威性审计；
9. 执行安全边界审计；
10. 执行验证绑定审计；
11. 生成完整 HTML；
12. 在 HTML 末尾写入 Known Limitations / TODOs。

不得在最终输出中展示中间推理。所有审计结果都应体现在文档内容中。

---

## 20. 质量验收标准

最终 HTML 必须满足以下条件：

- 新接手者能理解项目目标和非目标；
- 新接手者能按 Quickstart 跑通最小闭环；
- 维护者能定位核心模块的输入、输出和失败行为；
- 审计者能解释任意关键规则为什么成立或不成立；
- 操作者能知道哪些命令会写库、访问网络、消耗凭据或产生敏感数据；
- 代码智能体能从文档中获得项目结构、修改约束、测试命令和 done 标准；
- 文档中不存在未定义术语；
- 文档中不存在并列对象只展开一部分的情况；
- 文档中不存在无证据的“已验证”；
- 所有未知项集中出现在 Known Limitations / TODOs；
- HTML 离线打开无外部资源请求；
- 页面结构利于检索，而不是依赖读者从长文中找信息。

若无法满足某项，必须在 Known Limitations / TODOs 中说明原因和补齐所需材料。

---

## 21. 可直接追加给模型的任务语句

在实际使用时，可以在项目材料之后追加以下任务语句：

```text
请根据上述项目材料和提示文档，生成一份完整 self-contained HTML 项目文档。

要求：
1. 不要润色已有文档，而是重构为开发者文档站点风格。
2. 最终只输出完整 HTML 源码，不输出 Markdown、解释、前言或后记。
3. HTML 必须离线可打开，不依赖任何外部资源。
4. 必须包含 Overview、Quickstart、Architecture、Core Concepts、Workflows、Rules and Decision Logic、Data Model、Configuration、Commands and Reference、Validation、Failure Modes and Troubleshooting、Security / Privacy / Operational Boundaries、Maintenance、Agent Instructions、Known Limitations / TODOs。
5. 必须执行术语闭包、对称展开、证据绑定、命令副作用、数据权威性、安全边界审计。
6. 对输入材料中无法确认的事实，使用 UNKNOWN / TODO / UNVERIFIED，不得编造。
7. 用户指出的问题只是症状，必须全文查找同类问题并统一修复。
```

## 22. 文档修改任务固定前缀

当任务不是从零生成，而是修改已有 HTML 文档时，在每次请求前追加以下前缀：

```text
你接下来不是做局部润色。用户指出的问题只是一个实例。你必须先全文搜索同类问题，再统一修改。

本次修改必须执行以下全局检查：

- 是否存在被命名但未定义的术语；
- 是否存在并列列举但只展开其中一部分；
- 是否存在规则编号、测试项编号、状态标签没有完整定义；
- 是否存在“已验证 / 支持 / 高置信 / 只读 / 安全 / 权威”等结论但没有证据；
- 是否存在模块只有路径没有输入输出；
- 是否存在命令只有命令行没有副作用；
- 是否存在数据对象没有身份键、覆盖语义或权威性说明；
- 是否存在安全边界只写原则没有落到具体行为；
- 是否存在前后章节对同一概念使用不同名称；
- 是否存在辅助工具被误写成正式能力。

你必须修复全文中的同类问题，而不是只修用户点名的一处。若无法从输入材料确定事实，使用 UNKNOWN 或 TODO: source missing，禁止编造。
```

## 23. 审计优先、生成其次的两轮流程

对于较大项目，推荐不要一次直接生成最终 HTML，而是采用两轮流程。

第一轮任务：

```text
请只做文档审计，不生成最终 HTML。基于项目材料，输出以下内容：
1. 已确认事实；
2. 推断但未验证的内容；
3. 输入材料中的冲突；
4. 未定义术语；
5. 非对称展开的规则、模块、命令、数据对象；
6. 无证据结论；
7. 缺失的 Quickstart 信息；
8. 缺失的配置、命令、数据模型、验证、排错、安全边界；
9. 必须向用户确认或必须从源码补证的内容。

不要修改文档，不要生成 HTML。
```

第二轮任务：

```text
请根据项目材料、上一轮审计结果和本提示文档，生成最终 self-contained HTML。若审计中存在无法补齐的信息，在对应章节和 Known Limitations / TODOs 中显式标注，不得编造。
```

---

## 24. 最终提醒

项目文档的质量不取决于文字是否流畅，而取决于它是否让项目可接手。不要用摘要替代定义，不要用目录树替代契约，不要用“已验证”替代验证方法，不要用“只读”替代具体行为，不要用漂亮 HTML 替代工程事实。文档必须把项目的目标、状态、结构、规则、数据、命令、边界、失败和验证全部显式化。
