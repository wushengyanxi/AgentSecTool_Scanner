AgentSecTool · OpenClaw 暴露面普查 · 研判机制说明

OpenClaw 网关暴露探测的研判机制与版本取证

Premise

本文档说明扫描器如何对一个开放端点判定其是否为 OpenClaw 网关、并在判定为真时确定其运行版本。判定不采用概率打分，而是以一组按对 OpenClaw 身份的指示性分级的观测信号，经一份明确的证据组合白名单作二元裁决；版本取证则区分服务端自报的直读与基于前端资产指纹的反推。全文每一条机制均对应探测器源码 prober/openclaw/，可逐条复核。

01

判定为何是二元裁决，而非置信度打分

对暴露面普查而言，一个目标"是不是 OpenClaw"是一个需要被负责任地呈现的二元事实，而非一个介于 0 与 1 之间的分数。本节交代这一设计立场及其前提。

面向公网的服务指纹识别长期采用加权打分：为每个特征赋一个权重，命中即累加，超过阈值则判为目标。打分法在每个特征都足以单独指示目标身份时是合理的；但对一个控制面板类的目标，它有两重失效。其一，弱特征并非目标所独有——首页标题、站点图标、若干安全响应头等，其它服务也可能恰好具备，把它们相加得到的高分并不代表高可信，反而容易将非目标服务误判为目标。其二，分数本身不可解释：一个 0.82 的目标与一个 0.79 的目标在运营处置上意味着什么差别，无从回答。

本扫描器改以证据指示性分级加组合白名单替代打分。每个观测信号先按"它在多大程度上是 OpenClaw 运行时所独有"归入确证级、强证据、弱证据三档；随后由一份穷举的白名单规定哪些组合才足以判真，其余一切组合一律判假。判定结果因此是二元的，且每一次判真都能追溯到它满足了哪一条白名单规则——这正是普查数据对外披露时所要求的可问责性。

判定不变量

判定全程只读。HTTP 侧仅发起无鉴权 GET；WebSocket 侧在整个连接生命周期内只执行一次写操作——即升级握手请求本身，绝不发送任何 WebSocket 数据帧。探测不触发目标的任何状态变更。

02

对单个目标实际执行的探测序列

对每个待测端点（默认端口 18789），扫描器执行一组固定的只读探测，分布在 HTTP 与 WebSocket 两个协议表面。这一"跨表面"的设计是后续白名单的基础。

单目标探测序列：四次 HTTP GET 与一次 WebSocket 升级握手并行采集观测，汇入 evaluate() 作白名单裁决。两个协议表面（HTTP 路由面、WS 协议面）的分离，是 C2 规则"跨表面双强"的物理依据——见 §04。

七个测试项中，GET / 一次请求同时支撑三项观测（首页标题 T6、三项安全响应头 T7，以及用于版本反推的前端资产哈希），故请求数为五而测试项为七。每个测试项的完整请求与响应原文均被截断入库（响应体上限 8 KiB），供事后复核与争议追溯。

04

判定白名单：C1 与 C2

判真当且仅当满足以下两条规则之一，其余一切证据组合——包括三个弱证据的任意叠加——一律判假。本节给出两条规则及其设计动机。

判定决策。黑色终端节点表示一个被负责任地对外断言的二元结论——数据落地于此、不再流转。C1 走确证级单条达标，C2 要求两个独立协议表面各出一个强证据、互为佐证，二者取或。三个弱证据无论如何叠加，都不进入这张图的任何一条达标路径。

C1：确证级单条达标

当 control-ui-config 返回 200 且自报 serverVersion（即 T1），单条即判真。其正当性在于该信号是 OpenClaw 运行时所独有：能在此端点返回结构正确、携带版本号的 JSON，本身即表明运行的是 OpenClaw 该端点的逻辑，其它服务不会恰好产生这一响应。对确证级证据，单条命中即足以定案，无须第二个信号佐证。

C2：跨表面双强互证

当 WS 首帧命中 connect.challenge（T2），且 HTTP 路由面命中 T3 或 T4 之一时，判真。这条规则的关键不在"两个强证据"这个数量，而在它们分处两个独立的协议实现表面：T2 是 WebSocket 协议交互，T3 与 T4 是 HTTP 路由行为。非 OpenClaw 的服务即便在某一表面与之偶然相似，也极难在 WebSocket 协议面与 HTTP 路由面同时呈现 OpenClaw 特有的行为。跨表面的要求，把"仅凭单一表面的偶然相似即判真"这一最易产生误判的情形排除在达标之外。

弱证据的地位

T5、T6、T7 三个弱证据可被任意第三方廉价复制，因而不参与任何达标规则。一个同时命中全部三个弱证据、却不满足 C1 或 C2 的目标，仍判为假。它们只被收入 matched 列表与报告，用于人工复核与未来调参，绝不单独支撑"是 OpenClaw"的对外断言。

05

版本取证：直读、指纹反推与不可区分区间

判真之后确定运行版本。取证分两条途径：服务端自报的直读，与基于前端资产指纹的反推。反推须如实区分"精确到单版本"与"只能定位到区间"两种情形，不得谎报精度。

来源
机制
呈现
direct
T1 已直读到 serverVersion 字段。去除可能的前导 v 后即为版本。
精确版本，最高可信。
implicit
未直读到版本，转而以首页前端资产（生产构建的内容哈希文件名，如 index-DC4lnoz-.js）的集合与指纹库精确匹配，命中唯一版本。
精确版本。资产哈希集合相等才算命中。
implicit-range
资产指纹同时匹配指纹库中多个相邻版本——这些版本仅有后端改动、前端构建产物完全相同，外部不可区分。
版本区间，附全部候选。不谎报为单一精确版本。

指纹库由 fingerprint/build_corpus.py 逐版本起容器、记录其外部可观测签名构建而成。反推的判据是前端资产哈希集合相等，而非子集包含——内容哈希文件名随每次构建变化，集合相等才表明前端构建产物完全一致。

为何不用 CSP 哈希回退反推

control-ui 返回的那串 Content-Security-Policy 是静态常量、全版本相同，区分力为零。若以它回退匹配，会命中指纹库的全部版本，产出一个毫无意义的"全版本区间"。因此资产哈希未匹配时，扫描器如实记为版本未取到，而不退化为一个貌似携带信息、实则无区分力的版本区间。指纹反推所用的资产哈希采集自 GET / 这一步，与 control-ui 端点无关。

06

判假与探不到：失败原因的区分

一个目标可能因不是 OpenClaw 而判假，也可能因根本探不到而判假。当判假、无任何测试项命中、且 HTTP 侧无响应时，扫描器补做一次 TCP 连接以区分失败原因，使"非目标服务"与"不可达"不被混为一谈。

分类
含义
down
TCP 端口开着，但探不出任何 OpenClaw 证据——端口上跑着别的服务，非目标。
connection_refused
端口关闭，连接被拒。
timeout
连接超时——主机不响应，或被链路丢弃。
unreachable
网络不可达 / 无路由。

分类逻辑见 probe.go 的 classifyReachability()。这层区分对普查统计很重要：把"端口开着但非 OpenClaw"与"主机不可达"分开，才能正确估计暴露面的真实规模。

07

已知局限

基于外部观测的判定有其固有的能力边界。如实陈述这些边界，是普查数据负责任披露的一部分。

  • 弱证据不独有。站点图标、首页标题与常见安全响应头并非 OpenClaw 运行时所独有，其它服务也可能恰好具备。若仅凭这些特征即判真，会把非目标服务误判为 OpenClaw，故它们不参与白名单达标——这是为降低误判而付出的设计取舍。
  • 宁可漏判，不可错断。相应地，若某个真实 OpenClaw 实例恰好屏蔽了 WebSocket 协议面与两个特异 HTTP 端点、只暴露静态首页，则会被本机制漏判。系统在「漏判」与「误判」之间，明确偏向前者，以保证已收录资产的可靠性。
  • 区间不可消解。相邻版本若仅有后端改动、前端构建产物相同，则外部无法区分，只能给出区间。这是观测能力的固有上界，非实现缺陷。
  • 指纹库覆盖。未被采集进指纹库的新版本，其资产哈希不命中，版本报为空——判真不受影响，仅版本待补。
