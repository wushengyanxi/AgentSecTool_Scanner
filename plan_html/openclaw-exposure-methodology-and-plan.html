AgentSecTool Scanner · 测量方法与开发计划

在公网上发现暴露的 OpenClaw 网关，完成版本指纹，并据此做协同披露

前提 · Premise

目标是构建一个暴露面（exposure）测量管线：在公网范围内找出对外开放的 OpenClaw 网关，判定它确实是 OpenClaw 而非误报，再尽可能确定其版本，最终把结果汇成一张可下钻的暴露面板，并驱动对运营者的协同披露。这是测量与披露，不是利用——探测全程只读、良性，绝不触碰会改变目标状态或越权访问的接口。

之所以说"教科书式的单端口扫描"在公网做不到，原因不是一条而是六条：覆盖不全、吞吐不够、无法区分、被前置代理遮蔽、噪声与漂移、以及会被封禁并踩到合规红线。下面把每一条的对策拆开，再给出一份可独立推进的分阶段计划。

01

为什么公网上"教科书式扫描"做不到

把六种失效点逐一摆出来：每一种都说明朴素做法（nmap -p 18789 跑几段地址）为什么在这一点上失败，以及本管线如何应对。这是后面所有设计的出发点。

失效点
朴素扫描为何失败
本管线的对策
覆盖
只扫 18789 会漏掉三类主机：被反向代理放到 80/443/8080 的、改到自定义端口的、以及只在 IPv6 上可达的。再加上几段手选地址，覆盖面相对全网可忽略不计。
全 IPv4 无状态端口探测（端口集而非单端口）＋ 被动情报种子（已索引 OpenClaw 的测绘平台、证书透明日志、被动 DNS）＋ IPv6 命中表。三路并取以补齐盲区。
吞吐
对 37 亿个 IPv4 地址逐个做 TCP 连接级扫描、再逐个完成应用层握手，在可接受的时间内不可行；nmap 的连接扫描量级差了几个数量级。
两阶段：先用 masscan/ZMap 做无状态 SYN 探测快速圈定"端口开放"的主机，再仅对这批响应主机做代价较高的应用层指纹探测。这是 Censys 式架构。
区分
端口开放不等于跑着 OpenClaw。18789 上可能是别的 WebSocket 服务、是蜜罐、或是随手占用。仅凭开放端口判定会产生大量误报。
应用层正向指纹：发起 WebSocket 升级并观察 OpenClaw 特有的 connect 握手协商语义；多信号交叉验证（≥2 个证据），把误报压到极低。
前置遮蔽
CDN、WAF、反向代理挡在前面时，你指纹到的是代理而不是源站，版本无从谈起；朴素扫描对此毫无察觉，直接把代理当成目标。
对被代理实例改走可穿透的信号：仪表盘静态资产哈希、favicon 哈希、版本门控端点行为往往随响应透传；并标注"经代理、源站版本不可见"的置信状态。
噪声与漂移
动态 IP、临时主机、GeoIP 误差、丢包导致的漏计，会让"一次快照"既高估又低估，且无法去重——同一实例换了 IP 就被算成两台。
按固定周期复扫并做时序对账；按稳定身份（TLS 证书、JARM、内容哈希）去重，IP 仅作弱标识；对每个判定给出置信度而非非黑即白。
封禁与合规
高速朴素扫描会被防火墙拉黑、被滥用举报、并可能触法。被封既是合规问题，也直接污染数据——被拦截就意味着漏计。
公民化扫描基座：可识别的扫描源、按 /24 限速、尊重 opt-out 黑名单、只探测不利用。既守住合规边界，也换来更干净的测量结果（见 06）。
02

目标画像：OpenClaw 对外可观测的指纹面

能不能做版本指纹，取决于 OpenClaw 在网络上暴露了什么。把已知的可观测面集中列出——其中协议握手与版本方案是两个最有价值的抓手，而具体的"信号到版本"的映射需要在阶段 0 用真值语料实测确认。

服务
自托管 AI 智能体编排网关；可驱动邮件、浏览器、终端、日历与即时通讯。一旦无鉴权暴露，等价于把高权限的远程执行能力开放到公网，这正是它值得被测量与披露的原因。
默认端口 / 协议
TCP/18789，其上为 WebSocket 服务，同端口亦提供 HTTP 仪表盘；运营者常把它迁移或反代到 80/443/3000/8080 等端口。
默认绑定
默认仅绑定 127.0.0.1。因此公网可达的实例本质上是"绑到了 0.0.0.0 且未开鉴权"的配置错误——暴露面几乎全部来自误配置。
握手协议
所有客户端经 WebSocket 连接，首帧为 JSON 文本的 connect 请求，声明 minProtocol/maxProtocol（协议版本区间）、角色、scope 与设备鉴权（公钥＋签名）。协议版本协商是强指纹抓手，但只读：本管线只观察协商，绝不发送会改状态的帧。
版本方案
CalVer，形如 2026.1.29 / 2026.2.15 / 2026.3.13。便于做"是否脆弱"的区间比较，也便于把版本直接映射到 CVE。
关键脆弱性
CVE-2026-25253 鉴权令牌窃取致 RCE，影响 < 2026.1.29，CVSS 8.8。
CVE-2026-25593 经 WebSocket config.apply 的未授权本地 RCE——正因如此，探测器永远不会发送 config.apply，只需握手即可指纹。
CVE-2026-32922 提权类。版本→CVE 的映射作为面板里"脆弱占比"的判据。
已知暴露规模
据 2026-02 公开披露，彼时已发现 40,000+ 暴露实例，约 63% 被评估为可被远程利用。这类公开数字本身就是粗略估计——把它做准，正是本管线的价值所在。
03

测量管线总览

整条管线是一条数据流：主动与被动两路种子汇入候选队列，候选主机经只读的应用层指纹探测后进入版本预言机，给出版本与置信度；结果去重并写入时序存储，最终成为暴露面板与披露动作。公民化基座是贯穿全程的约束。

方框为管线阶段，箭头为数据流向。上方"主动 / 被动"两源并入候选队列；只读探测把候选交给版本预言机，输出版本与置信度，再按稳定身份去重后入库。最下方虚线条带是贯穿全程的合规约束，不在数据流内。深色的终端框是系统出口——测量在此转化为面板与披露动作。版本→CVE 一行展示了 CalVer 比较如何直接产出"脆弱占比"。

04

发现：在全网范围内把候选主机找全

发现阶段只回答一个问题：哪些主机值得做下一步的应用层探测。把这个答案做全，需要主动、被动、IPv6 三路并取——任何单路都有结构性盲区。

主动发现负责广度。用 masscan 或 ZMap 做无状态 SYN 探测，对一个端口集而非单端口扫描：以 18789 为主，叠加 80/443/3000/8080/8443 等常见反代落点，再带一条常见替代端口的长尾。无状态扫描能在受控速率下于较短时间内覆盖全 IPv4 的单个端口，这是连接级扫描达不到的量级。它的产物只是"端口开放"的主机清单——粗、但全。

被动发现负责那些主动扫描看不见或不该去敲的主机。已索引 OpenClaw 指纹的网络测绘平台（Censys、Shodan、FOFA、ZoomEye）可直接给出候选；证书透明（CT）日志能捞出带 OpenClaw 特征的主机名与证书；被动 DNS 与 rDNS 的 PTR 命名模式可作补充。被动种子更便宜、更轻扰，也能覆盖到藏在反代后、主动扫描只会指纹到代理的实例。

IPv6 不能靠暴力遍历——地址空间过大。只能依赖命中表（hitlist 服务）、被动源与 CT/DNS 推导出的活跃地址集。把 IPv6 当作"尽力而为"的覆盖，并在面板上如实标注其覆盖局限，而不是假装它和 IPv4 一样完整。

三路的产物统一汇入候选队列，去抖、按 /24 限速后，逐条交给指纹探测。这一层与判定层解耦：发现可以激进求全，因为真正定性的把关在下一层。

05

判定与版本预言机

这一层把“端口开放的候选”收敛为“确实是某版本 OpenClaw 的实例，置信度为 X”。它由两件事构成：高精度的判定，和一个把可观测信号翻译成版本的预言机。

判定要把误报压到极低，因为暴露面板的每一个计数都会被引用、甚至驱动对运营者的通知，误报的代价很高。做法是正向指纹加多信号交叉：发起 WebSocket 升级，观察目标是否按 OpenClaw 的方式协商 minProtocol/maxProtocol、是否对一个最小化的良性 connect 帧给出 OpenClaw 特有的握手/错误语义；同时核对 HTTP 仪表盘侧的特征（标题、JS 资产路径、favicon 哈希）。要求至少两个独立证据吻合，才判定为 OpenClaw，以此抵御蜜罐、无关 WebSocket 服务与反代造成的假阳。

探测必须只读、良性。这是不可逾越的界线，也是判定设计的硬约束：握手协商足以识别服务与协商出的协议版本，因此探测器止步于握手，绝不发送任何会改变目标状态或尝试越权的帧——尤其绝不触碰 config.apply（其本身即 CVE-2026-25593 的利用面）。识别一台脆弱主机，不需要、也不允许去利用它。

版本预言机分显式与隐式两条路。显式：connect 协商出的协议版本区间映射到发布窗口，或服务在握手响应、HTTP 头、仪表盘资产里直接带出构建/版本串。隐式（当没有显式版本时）：仪表盘 JS/CSS 资产哈希、favicon 哈希、版本门控端点的存在与状态（例如某端点在特定版本上返回 404 而在另一版本可用）、错误文案措辞、所支持的协议版本集合、被代理时的 TLS JARM/JA3S 指纹。把这些信号维护成一张指纹库，输出的是具体版本或版本区间＋置信度，而非伪精确的单一版本号。

指纹库不能凭空写，必须由真值语料标定：在受控环境（Docker）里跑起已知版本的 OpenClaw，记录它们从外部可观测的全部签名，作为指纹与版本的对照基准——这也是阶段 0 的核心产物。最后，按 CalVer 把版本映射到 CVE（< 2026.1.29 ⇒ CVE-2026-25253，等等），直接产出面板上的"脆弱占比"。

06

公民化与合规：把"做得到"和"做得对"绑在一起

这一节不是免责声明，而是设计的一部分。一个隐蔽、激进的扫描器会被封、被举报、被漏计，数据反而更差；一个可识别、克制、尊重 opt-out 的扫描器既守住合规，也拿到更干净的测量结果。两个目标在这里是一致的。

  • 可识别的扫描源。扫描 IP 配 rDNS、在该 IP 上挂一个说明研究目的与退出方式的网页、提供 abuse 联系人。让被扫方一查即知"这是谁、为什么、如何退出"。这与"规避检测"恰恰相反——研究式扫描要响亮、要可问责。
  • 尊重 opt-out。维护并尊重退出黑名单与已知的不可扫描网段；收到退出请求即从后续扫描中剔除。
  • 限速以不伤目标。按 /24 控制速率，避免对任何单一网络造成可感知的负载。测量的目的是观察，不是压测。
  • 只探测，不利用。全程止于识别与版本指纹；不做认证、不改状态、不访问数据、不发利用载荷。这条与 05 的探测约束是同一条线。
  • 合规边界。明确授权与法律基础（学术测量场景、检测而非入侵），遵循目标司法辖区与本地法律（如《网络安全法》等），并尊重平台服务条款。检测—披露是稳妥车道，利用不是。
  • 协同披露。发现转化为通知，而非公开点名：经 abuse 联系人、国家级 CERT 或托管商，给运营者留出修复窗口；面板对外只呈现聚合统计，不公开可定位到个体的暴露清单。

凭据安全：测绘平台（Censys/Shodan/FOFA 等）的 API key、扫描基础设施凭据一律走配置文件、运行时读取，绝不硬编码进代码；配置文件纳入 .gitignore。

07

开发计划：分阶段里程碑

这是一个从零起步的项目，因此计划按"先把定性做对，再把规模做大"排序。每个阶段都附一条可验证的成功标准——标准够强，阶段之间才能独立闭环推进，而不必反复回头确认。

阶段 0
地基

指纹与真值语料

在 Docker 里跑起若干已知版本的 OpenClaw，记录它们从外部可观测的全部签名（WS 握手协商、HTTP/仪表盘资产、favicon、版本门控端点行为），据此搭出指纹库的第一版与一套精确率/召回率评测脚手架。这是成败所系的一步——下游全部依赖一个高精度、能区分版本的指纹。

验证：对标注语料，判定精确率 ≥ 0.99、召回 ≥ 0.90；版本（或版本区间）判对率达成既定门槛。

阶段 1
探测器

单点探测器 MVP

输入一个 IP:port，输出 {是否 OpenClaw, 版本/区间, 置信度, 证据}。只实现只读、良性探测路径。先把定性这件事在单点上做扎实，再谈规模。

验证：在阶段 0 真值集上复现既定的精确率/召回；对良性帧之外的任何请求有硬性禁用开关。

阶段 2
发现

主动＋被动发现接入

接入 masscan/ZMap 无状态端口探测与被动种子（测绘平台、CT、被动 DNS），把候选 IP:port 喂给阶段 1 的探测器；IPv6 走命中表尽力覆盖。

验证：对一段已知存在暴露实例的受控/授权地址范围，端到端跑通并复核计数；被动源与主动源的去重正确。

阶段 3
规模

调度、去重与时序存储

周期性全量复扫；按稳定身份（TLS 证书/JARM/内容哈希）去重，处理资产漂移；把"主机 × 版本 × 暴露状态 × 时间"写入时序存储，支撑趋势与下钻。

验证：同一实例换 IP 后仍被识别为同一资产；两次扫描间的新增/消失可解释。

阶段 4
产出

暴露面板＋协同披露

面板呈现版本分布、各 CVE 的脆弱占比、按国家/ASN 的分布与时间趋势，可下钻；对外只给聚合统计。披露侧走协同流程，经 abuse 联系人/CERT 通知运营者并尊重 opt-out。

验证：面板数字与底层存储一致可对账；披露流程在内部演练中可走通且不外泄个体暴露清单。

贯穿 / 可选

公民化基座与披露自动化（持续）

扫描源身份、opt-out、限速作为贯穿全程的基座先于规模化落地；披露通知的自动化与 IPv6 的更高覆盖作为可选增强，按授权范围与资源推进。

验证：每次规模化扫描前，基座的可识别性与限速生效；凭据均来自 .gitignore 的配置而非代码。

待定决策

决定于 · 阶段 1 之内

这份文档把"在公网上测量 OpenClaw 暴露面"拆成了可独立推进的几步：先用真值语料把指纹与版本判得准，再用两阶段的主动＋被动发现把覆盖做全，最后在一个可识别、克制、尊重退出的扫描基座上把结果汇成面板并驱动协同披露。贯穿始终的判断是：识别一台脆弱主机以促成修复，既不需要、也不允许去利用它——这条线既是合规边界，也是把数据做干净的前提。
