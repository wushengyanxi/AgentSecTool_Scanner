面向公网的 OpenClaw 网关暴露普查系统

系统汇报文档 · 中国大陆范围 · 检测优先

摘要

OpenClaw 是一类带 Web 控制面与 WebSocket 控制协议的 AI 网关，其管理端口暴露于公网且未启用鉴权时构成可被利用的攻击面。本文介绍一套面向中国大陆公网的 OpenClaw 暴露普查系统，它以只读方式发现候选主机、可靠判定其身份、并尽力确定运行版本，为负责任的漏洞披露提供可追溯的资产底账。系统按数据流向组织为发现、探测、落地、呈现四层，辅以版本指纹工具链与块级扫描进度管理。其设计有三个要点：探测全程只读，不触发目标的任何状态变更；身份判定以按特征对 OpenClaw 的指示性强弱分级的观测证据经二元白名单裁决，使每一次判真都可回溯到所满足的规则与原始请求／响应；第三方平台凭据一律不入代码。系统目前已完成端到端闭环，指纹库覆盖 88 个 OpenClaw 版本，被动测绘累计约八万条 FOFA 候选与十六万行 ClawSec 快照，已收录确认资产 5762 台。

1引言与设计目标

网络空间中暴露的管理控制面是一类长期存在的安全隐患。OpenClaw 作为带运行时控制能力的 AI 网关，一旦其默认端口（18789）在公网可达且未设鉴权，攻击者即可直接接管其控制协议。对这类暴露做一次系统性的普查，是开展负责任披露、推动修复的前提。本系统正是为此而建，其总体目标可概括为：在中国大陆地址空间内发现 OpenClaw 暴露候选，对每个候选作可靠的身份判定，并在判定为真时确定其运行版本，最终形成一份可对外问责的资产底账。

本系统在设计之初即确立了三条贯穿始终的原则，它们体现在后文将要讨论的每一个子系统中。其一是检测优先。探测全程只读：HTTP 侧仅发起无鉴权的 GET 请求，WebSocket 侧在整个连接生命周期内只执行升级握手这一次写操作，绝不发送任何控制数据帧。系统不向目标提交任何会改变其状态的指令，普查行为本身不构成对目标的干扰。其二是判定可问责。身份判定不采用业界常见的加权打分，而是把观测信号按其对 OpenClaw 身份的指示性强弱分级，再由一份穷举的证据组合白名单作二元裁决。这样得到的结论是确定的「是」或「否」，且每一次判真都能指明它满足了哪一条规则、依据了哪些原始观测——这正是普查数据对外披露时所要求的可追溯性。其三是凭据零硬编码。系统所依赖的第三方测绘平台凭据，均经环境变量或被版本控制忽略的本地配置文件读取；真实暴露主机清单不进入代码仓库。

本文余下部分的组织如下。第 2 节给出系统的总体架构与分层间的数据契约；第 3 节至第 6 节依数据流向，分别论述发现、探测、落地与呈现四层的设计与实现；第 7 节讨论系统的已知局限与后续工作。系统的身份判定与版本取证机制涉及较多细节，本文只作概述，其完整论证另见《OpenClaw 网关暴露探测的研判机制与版本取证》。

2总体架构

系统按数据的自然流向划分为四个层次。发现层在公网范围内产出 OpenClaw 暴露候选，既包含对第三方测绘平台的被动检索，也包含对给定地址范围的主动扫描。探测层对每个候选执行只读探测，判定其身份并取版本。落地层将探测结果按可信级别分类后规整入库，并完整留存每次探测的请求与响应。呈现层实时查询入库数据，将收录资产与逐主机的研判解读呈现给使用者。除这条主线外，系统还有两个支撑子系统：版本指纹工具链为探测层的版本反推提供依据，块级进度管理则为大规模主动扫描提供可中断、可协作的调度。图 1 给出系统的整体数据流。

图 1系统数据流。中部为左至右的主流水线，下方为按来源分置的数据层，上方为支撑工具链。实线表示数据流动，虚线表示支撑关系；黑色终端节点表示面向人的最终呈现。

这一分层的意义不止于概念上的划分。各层以稳定的中间产物相衔接——发现层产出候选清单，探测层产出带完整请求与响应的探测结果，落地层产出结构化的资产底账——从而得以各自独立演进，任一层的实现可以替换而不波及其余层。这也使得以 Go 实现的探测核心与以 Python 实现的数据处理及呈现，能够在不同的编程语言与进程边界下协作。在存储上，系统按数据来源分库，使不同来源、不同生命周期的数据互不纠缠，而在分析确需跨源关联时于查询层临时连接，无需引入任何外部依赖。

3核心模块与功能

本节依数据流向，逐一分析构成系统的核心模块——它们各自承担一个明确的职责，并以一以贯之的设计思路实现该职责。发现层广撒网产出候选，探测层以分级证据作可问责的判定，落地层按可信级别沉淀资产，呈现层把结论还原为可核验的证据。四者衔接成从「公网上可能有什么」到「确认了什么、依据为何」的完整链条。下文不纠缠于具体接口形态，而着眼于每个模块「按什么思路做了什么」。

3.1发现层：被动测绘与主动扫描

发现层的职责是产出尽可能完整的 OpenClaw 暴露候选集合。它综合了两类互补的发现途径：被动途径从已有的网络空间测绘平台获取已知暴露主机，主动途径则在指定地址范围内自行扫描开放端口。两者在覆盖面上各有所长——被动源能即时给出测绘平台已收录的实例，主动扫描则能触及平台尚未覆盖的暴露面。

被动测绘有两个独立的来源，其取数思路各自呼应了来源的特性。FOFA 子系统检索中国大陆范围内的候选；鉴于平台查询额度受限而本地存储充裕，它采取「一次性、富字段、全量拉取并原样留存」的思路，把候选连同其原始信息一并存下，避免日后为同一数据重复消耗额度。ClawSec 子系统对接一个公开、频繁更新且自带版本判定与漏洞标注的暴露面测绘平台。针对其逐日更新的特性，它采取「每日全量快照」的思路，以平台公布的更新时间为归档时间戳，同一实例每日留存一份。这一思路的价值在于使纵向分析成为可能：长期持续在线的实例可被识别出来，优先用于后续的真实地址枚举。系统亦保证每份快照的内部一致——若拉取途中平台更新到次日，便归档已得快照、转而重拉次日，并将中断者标记为不完整。

主动扫描由发现层的主动子系统承担，以多种扫描后端适应不同规模——从无须特权、面向小范围验证的连接扫描，到面向大规模探测的无状态扫描——其结果可与被动种子合并去重。扫描范围以中国大陆地址清单为基础，由权威的地址分配数据解析而成。面对全网规模，逐个地址记录进度并不现实，因此系统以「地址块」为单位、按「扫描批次」组织任务，支持原子领取、断点续扫与多进程协作，使一次覆盖大范围的普查可被中断并恢复。

3.2探测层：只读判定与版本取证

探测层是系统的判定核心，以独立于其余部分的语言实现，仅依赖标准库而不引入任何第三方包，从而保持探测核心的轻量与可移植。对每个候选，它执行一组只读探测——分布在 HTTP 与 WebSocket 两个协议表面——据以判定身份并尽力取版本，并完整保留每次探测的请求与响应交后续留存。这里的核心思路，是把判定建立在各项证据对 OpenClaw 身份的指示性之上，而非简单的特征加权。

具体而言，系统按一项特征「在多大程度上是 OpenClaw 运行时所独有」，把观测信号分为三级。最高一级是确证级，仅含一项：配置端点在运行时自报版本号——这一响应由 OpenClaw 服务端在运行时生成，其它服务不会恰好产生，因而它一旦命中即可单独定案。中间一级是强证据，包含服务端主动下发的协议交互事件，以及特异端点的行为特征；它们对应 OpenClaw 特有的协议行为与路由，普通服务极少与之巧合。最低一级是弱证据，即站点图标、首页标题与一组常见安全响应头——这些内容并非 OpenClaw 独有，其它服务也可能恰好具备，单独不足以作为身份依据。

在这一分级之上，判定由一份二元白名单作出，而非加权累加。系统判真当且仅当满足以下两条规则之一：其一是确证级证据单条成立；其二是 WebSocket 协议面的强证据与 HTTP 路由面的强证据同时成立。后一条规则的要义不在「两个强证据」这个数量，而在它们分处两个独立的协议实现表面——非 OpenClaw 的服务即便在某一表面与之偶然相似，也极难在两个独立表面同时呈现 OpenClaw 特有的行为。三个弱证据无论命中多少，都不参与达标。这一设计把「仅凭非独有的静态特征即判真」这一最易产生误判的情形，明确排除在外。

判真之后，系统尽力确定版本。版本取证分两条途径。若确证级端点已直接给出版本号，则直读，结果最为可信。否则，系统转而以首页前端资产（即生产构建产生的、文件名带内容哈希的 JS 与 CSS）的哈希集合，与版本指纹库精确匹配，反推版本。当资产指纹同时匹配指纹库中多个相邻版本——这些版本往往仅有后端改动、前端构建产物完全相同，外部无从区分——系统如实给出版本区间与全部候选，而不谎报为单一精确版本。这一克制是有意为之：当资产哈希未匹配任何已知版本时，系统老实记为版本未取到，而非退化为一个貌似携带信息、实则毫无区分力的「全版本区间」。

支撑版本反推的指纹库，由专门的采集工具链逐版本构建而成，当前覆盖 88 个 OpenClaw 版本。其中前端资产的内容哈希是版本的主区分信号，而图标等跨版本稳定的特征仅用于身份判定、不参与版本区分。

3.3落地层：分类入库与时序留存

落地层将探测结果按可信级别分桶后规整入库。其分类口径直接服务于「实例收集」这一目标：明确判真且取到版本者，记为已确认；判真但版本待定者，记为确认但版本未取到；中了部分特征却未达白名单者，记为疑似——这类记录保留下来供日后复扫与扫描策略优化，但不会被呈现为 OpenClaw 实例；至于探测不到的目标，则不予收录。资产以「IP 与端口」为身份去重，每台独立暴露主机对应一条资产记录，这与「全网有多少台 OpenClaw 服务器」的统计口径相一致，而非按内容指纹跨地址合并。

沉淀下来的数据遵循「资产登记」与「时序观测」相分离的思路：一面为每台主机维护一条当前状态，一面为每次探测追加一条历史观测，并逐条留存每次观测中每个测试项的完整请求与响应原文。这一安排既支持「这台主机现在是什么状态」的查询，也支持「它历次被探测到的情况如何」的纵向追溯；而完整留存的请求与响应，使任何一次研判都可被事后复核与争议追溯。表 1 给出这三类记录的职责与当前规模。

3.4呈现层：本地看板与研判解读

呈现层将收录资产呈现为一个本地看板，含资产列表与版本、可信级别、版本来源的分布。其设计思路的关键，是把抽象的研判结论还原为可核验的证据。使用者展开任一资产，即可看到该主机的探测报告：报告逐个测试项陈述其请求、响应与观测事实，而把证据级别与达标逻辑收口于末尾的研判小结讲述一次。这种「观测在前、结论在后」的安排，使每个测试项只陈述自身这一步看到了什么，而不在局部重复全局的判定逻辑。

报告并非预先写死，而是按主机的真实响应填充生成，其中的关键字段皆取自该主机的实际响应。尤为重要的是证据的可见性：观测文案与请求、响应原文三处，对同一组关键特征统一标示，使读者得以将「研判文案中提到的特征」与「响应原文中的对应内容」相互印证。以版本指纹反推为例，报告会列出本次反推实际依据的前端资产，并使之与响应原文中的资产引用相互对应，从而让反推得出的版本结论有据可查，而非一句无凭无据的断言。

4局限与未来工作

基于外部观测的普查有其固有的能力边界，如实陈述这些边界是负责任披露的一部分。在覆盖面上，系统秉持「宁可漏判，不可错断」的取向，因而仅暴露静态特征、却屏蔽了协议交互与特异路由的实例会被漏判——这是为保证已收录资产的可靠性而付出的代价。在版本精度上，仅有后端改动、前端构建产物完全相同的相邻版本，从外部无从区分，只能定位到版本区间，这是观测能力的固有上界而非实现缺陷。在时效上，普查得到的是某一时刻的快照，而暴露面随实例上下线、版本升级而持续变化；尚未被采集入指纹库的新版本，其版本会暂时取不到，须待指纹库补充后方能确定。

后续工作主要围绕三个方向展开。其一是新版本的持续跟踪。OpenClaw 版本迭代频繁，每个新版本发布后须及时采集其指纹并入库，否则运行该版本的实例只能被记为版本未取到。持续维护指纹库、缩小这一比例，是保证版本判定时效性的前提，也是后续漏洞态势分析的数据基础。

其二是新增暴露实例的持续发现。暴露面并非静态——既有实例可能下线或修复，也不断有新的实例上线暴露。系统将把主动扫描与块级进度管理打通，以分批方式对中国大陆地址范围作周期性的全量普查，并与被动测绘源互补；同时借助测绘平台的逐日快照作纵向比对，识别出新近出现的暴露实例与长期持续在线的实例，使资产底账始终反映当前的真实暴露状况。

其三是整体漏洞态势的分析。在已收录资产的版本分布之上，将每台实例的运行版本与该版本已知的漏洞相对应，即可从全局视角评估各漏洞的影响面——哪些漏洞覆盖的暴露实例最多、风险敞口最大，哪些版本段集中了高危问题。这一态势视图为披露工作排定优先级提供依据：优先处置覆盖面广、危害严重的漏洞所对应的实例。系统最终将在这份可追溯的资产底账之上，依托完整入库的证据生成可核验的披露材料，推动暴露实例的修复——这正是整个系统存在的目的所在。

本报告所述模块职责、数据契约、表结构与统计规模，均取自系统当前实现与运行数据，可对照代码仓库逐项复核。系统判定机制的完整论证见《OpenClaw 网关暴露探测的研判机制与版本取证》，各类研判情景的真实用例见《研判用例样例报告》。
