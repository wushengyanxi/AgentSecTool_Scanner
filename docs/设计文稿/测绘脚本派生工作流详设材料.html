测绘脚本派生工作流详设材料

漏洞需求包进入测绘平台后，系统围绕代码、说明文档和补充材料建立可追溯的派生任务。任务中的智能体负责形成项目测试项、容器运行环境、验收项和展示模板；Harness 负责在隔离环境中产生可复查证据；审核通过的项目测试项以固定容器 worker 的形式进入能力仓。

系统边界

工作流处理的输入是上游漏洞项目交付的需求目录。目录中的代码和主文档构成主要分析材料，辅助文件提供环境、配置或运行线索。平台保存原始材料的不可变快照，并以独立补充包承接后续会话中上传的文件。原始包、补充包和候选文件记录 SHA-256；会话消息、工具事件、容器运行和审核结论记录时间与关联对象。

项目测试项用于观测目标实例是否满足某个项目级条件。一个需求可以派生多个测试项，多个漏洞也可以复用同一测试项。漏洞适用性由结构化规则组合项目测试项产生的事实，不由单个弱特征或单次模型判断直接确立。

能力入仓只注册已经验证的项目测试项及其容器镜像，不触发公网目标扫描。实际测绘任务由主扫描器在独立调度阶段启动。

需求包契约

每个需求目录包含一个 manifest.json、一个或多个 UTF-8 代码文件和一个主文档。主文档格式限定为 HTML、Markdown 或纯文本，并在正文中给出项目身份和漏洞或报告身份。YAML、配置、日志和样例数据可以作为辅助文件登记，但不承担固定语义。

{
  "schema_version": "1.0",
  "request_id": "request-2026-0001",
  "code_files": ["scanner.py", "helpers/protocol.py"],
  "document_file": "vulnerability-report.html",
  "auxiliary_files": [
    {"path": "environment.yaml", "kind": "environment_hint"}
  ]
}

导入器只读取 manifest 登记的文件。绝对路径、目录逃逸、符号链接逃逸、缺失文件和重复路径均导致校验失败。系统重新计算 SHA-256；相同 request_id 与相同内容 hash 返回已有任务，相同 request_id 与不同内容返回冲突。

智能体与持续会话

每个派生任务拥有一条持续会话。用户消息可以包含问题、修订意图、执行指令和附件引用。智能体在同一会话中调用读取材料、查询既有测试项、受控检索、写候选文件、构建镜像、运行 Harness、读取证据和形成候选能力等工具。工具事件与对话消息共同构成时间线。

任务状态由目标完成度决定。验收项通过并形成候选能力时，智能体结束当前运行；缺少事实、需要安全判断或无法取得新证据时，状态转为等待用户；基础设施不可用时记录执行失败。运行期间到达的新消息按顺序进入同一任务会话，并在当前运行结束后继续处理。

同一资产类型下的多个能力共享事实空间。智能体生成的事实字段需要使用稳定的项目命名空间，避免与同类其他测试项重名。多个能力引用同一漏洞编号时，其结构化适用条件必须一致；事实或规则冲突会阻止冲突结论进入确定状态。

工具输入边界输出
list_materials、read_material已登记原始包、补充包和当前 attempt 文件。材料清单与带 hash 的文本片段。
web_search配置允许的官方仓库、文档、注册表和可信镜像源。来源链接和检索结果。
define_acceptance_test当前任务中的验收项草案。带版本的待确认验收项。
begin_attempt、write_candidate_file当前 attempt 的安全相对路径。候选文件及 SHA-256。
run_harness候选环境、worker 和当前验收项。运行 ID 与逐项结果。
read_harness_run当前任务所属运行。日志、trace、命令和结构化结果。
register_candidate通过验证且验收项已经确认的测试项、规则和模板。候选能力包。
request_user_input无法由现有证据消解的具体问题。等待用户状态和问题说明。

测试项与验收项

项目测试项是生产测绘能力，负责在目标实例上观测一个条件。验收项是派生阶段的质量约束，负责判断项目测试项在指定样本和环境中的行为是否满足预期。二者分别保存，避免生产逻辑与自我验证逻辑混为一体。

智能体可以使用草案验收项开展探索。候选能力进入审核前，操作人必须确认所有启用验收项的阻断属性、断言和预期结果。智能体不能修改已经确认的验收项；操作人修订后形成新版本。验收样本数量由 Harness case 声明，不使用统一的正负样本数量。

字段含义
stable_key任务内稳定的验收项标识。
name、purpose验收项名称和验证目标。
enabled当前版本是否参与验收聚合；停用时必须记录理由。
blocking失败是否阻止候选进入审核。
assertion字段路径、操作符和预期值。
user_confirmed当前版本是否已经由操作人确认。
actor、change_reason版本形成主体及修改理由。

容器 worker 协议

审核通过的项目测试项以固定 OCI 镜像 digest 运行。主扫描器在扫描任务开始时启动长生命周期容器 worker，通过标准输入发送一行一个目标的 JSON，通过标准输出接收一行一个测试结果。Harness 使用相同协议验证候选镜像，保证验证对象与生产对象一致。

{"request_id":"r-1","target":{"host":"192.0.2.10","port":443,"tls":true},"timeout_ms":8000}
{"request_id":"r-1","test_id":"project.condition","status":"satisfied","facts":{},"evidence":[],"error":null}

状态限定为 satisfied、not_satisfied、unknown 和 error。每个能力包包含一个独立项目测试项，同一资产类型可以登记多个能力包。主扫描器调用相应的长生命周期 worker，聚合测试事实、版本和展示模板，并用结构化漏洞关联规则计算适用性。事实字段发生冲突时，冲突值单独保存，相关漏洞结果保持未知。

能力仓的 index.json 按资产类型保存能力数组。数组元素以 capability_id 区分独立项目测试项，并固定 manifest 路径和镜像 digest；同一 capability_id 的新版本替换运行索引中的旧版本，不影响同一资产类型下的其他项目测试项。

数据对象

对象职责关键关系

前端交互

派生工作台使用独立页面，与资产看板互相导航。操作人输入本机可访问的需求目录路径后，导入预览展示文件清单、hash、主文档和未登记文件。确认导入后进入任务详情，并通过任务级会话启动分析。任务看板显示当前阶段、最近工具事件、验收汇总和等待用户的具体原因。

任务详情同时呈现材料、验收项、attempt、Harness 证据、候选能力和持续会话。聊天输入支持文本与附件；附件先形成补充包，再与消息共同提交。失败卡点作为可展开事件显示，操作人可以引用事件继续质询。自然语言执行指令恢复同一任务，不提供单步重跑、整轮重跑或生成修订 attempt 按钮。

候选审核页显示已确认验收项、非阻断警告、固定镜像 digest、项目测试项、项目识别规则、漏洞关联、展示模板和来源证据。审核动作包括通过入仓和退回修改，动作理由与时间随候选记录保存。

资产看板按展示模板选择实例事实字段，并同时呈现逐项目测试项证据和漏洞适用性。OpenClaw 内置 detector 继续使用专用报告话术，动态能力不套用 OpenClaw 的测试项名称和判定说明。

安全与审计

模型生成文件只能写入当前 attempt 工作区，不能覆盖原始包、补充包和已入仓能力。Docker 构建与运行禁止 privileged、host network、宿主目录写挂载、Docker socket、设备映射和额外 capabilities，并设置 CPU、内存、进程数和超时。项目测试项不得在生产测绘中实施状态修改或漏洞利用。

OpenAI API key 只从 Git 忽略的本地配置读取。项目级授权控制需求材料是否可以发送给外部模型服务；智能体运行记录模型、状态、响应 ID、消息和工具事件，不保存 API key。Responses 请求使用 store=false，每轮请求回放完整输出项及加密 reasoning 内容，本地数据库承担任务会话与审计记录的持久化。
