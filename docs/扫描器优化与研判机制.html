OpenClaw 扫描器优化方案与研判机制

本文记录两件事

其一，扫描器以什么方式对一个目标做出有置信度的研判——不用任何拍脑袋的概率分值（没扫过全网，任何「95%」都不可辩护），而是陈述观测到的事实、按其可伪造性分级、再依一个明确的证据组合白名单做出二元研判（是 / 不是），并把支撑证据原样交给用户。其二，下一步怎么优化、以及它该长成什么样的命令行工具——对标 nmap 的 CLI / 分级日志 / 多格式报告，连同基于真实公网测试暴露的失效模式给出分梯队改进方案。所有结论以源码或真实容器/公网实测为据；实测否定的方向（如 WS 只读取版本）明确列为不做。

01

研判机制总览：从只读探测到有置信度的结论

扫描器对单个目标的研判分两层独立的判断：判定层回答「这是不是 OpenClaw」，靠证据分级 + 二元白名单（不靠概率分值）；版本层回答「是哪个版本」，用优先级回退的三条路径。两层都以「只读探测采集的证据」为唯一输入——只发 WebSocket 升级与无鉴权 GET，绝不发改状态帧。

深色框＝只读探测；三个彩框＝按「可伪造性」分级的证据（绿=确证、橙=强、灰=弱）；灰框＝由证据组合白名单做出的二元研判（True/False）；右框＝结构化结论，附原始证据清单供用户自查。判定与版本互不依赖：一台可「确认是 OpenClaw 但取不到版本」。

02

判定层：用证据分级，不用概率分值

为什么不给概率分值——因为没有人扫过全互联网，任何「favicon 命中＝95% 是 OpenClaw」都是拿一个脑补的分子除以一个未知的分母，谁都可以质疑。可辩护的做法是只陈述观测到的事实，并按一个客观标准给事实分级：这条证据能不能被静态抄袭/伪造。运行时才能产生、抄不了的是确证；高门槛但可被实现的是强证据；静态文件或常见特征、谁都能仿的是弱证据。

信号
为什么是这个级别（可伪造性）
证据级别
control-ui 200 + serverVersion
服务端运行时解析自身配置、把版本号塞进 JSON 响应。蜜罐要伪造得真去实现这个端点逻辑，抄静态文件抄不出来。
确证级
WS connect.challenge
连上 WebSocket 即被推送的协议事件（带新鲜 nonce/ts）。可伪造，但需伪造方主动实现 WS 服务端并模仿该协议行为——门槛高，不是抄文件。
强证据
control-ui 401
该特异端点存在、且行为符合（鉴权挡住）。要仿得实现这个端点。
强证据
/healthz 特征体
特定结构的健康检查响应。比静态文件难仿，但比协议行为易。
强证据
favicon MD5
静态文件 /favicon.ico。curl 下来原样挂上即可，纯抄袭。
弱证据
首页 title
静态 HTML 字符串，谁都能写 <title>OpenClaw Control</title>。
弱证据
响应头三件套
常见安全头，很多框架默认就设，判别力本就弱。
弱证据

最终结论是二元的（True / False）——证据分级只用来决定落到哪一边，不作为中间档位。判 True 由一个明确的证据组合白名单决定（完整推导与测试项编号见 §04）：

// True 当且仅当满足任一，其余一切组合 = False
C1:  T1                  // 确证级单独成立（control-ui 自报版本，伪造=真实现）
C2:  T2 && (T3 || T4)    // WS 强证据 + HTTP 路由面强证据，跨表面双强
  • 这套机制直接回应「分值太强」的质疑——过去 favicon 标 0.95、几乎和确证级一样高，等于默认没人会抄一个 favicon。新机制里 favicon 是弱证据，单独连 True 都判不了。一个抄了 favicon+title 的蜜罐，结论是 False（详细依据里如实列出命中项，并标「可伪造、疑似仿冒」）。
  • 跨表面冗余是 True 的硬门槛——最终结论是二元的（True/False，判据见 §04 的白名单 C1/C2），证据分级只用来决定落到哪一边。纯静态伪造（只有弱证据）一律 False；要判 True，必须命中确证级、或跨两个表面的双强证据。
03

版本层：三条路径与诚实的不确定性

版本判定独立于「是不是 OpenClaw」，走优先级回退。核心原则是能确定就给精确值，不能确定就如实给区间，绝不把猜测伪装成精确结论——这是真实公网验证中反复确认的正确行为。

优先级
路径
触发条件与可靠性
①
direct 直读
control-ui-config 返回 200 且响应含 serverVersion → 服务端自报版本，最可靠。前提：实例疏防（多见于 v2026.4.21 及更早，该端点自 v2026.4.22 起加鉴权）。
②
implicit 反推
直读不可得（401）时，用首页资产名集合（index-*.js/css，构建内容哈希）与全量指纹库精确匹配。资产名来自官方镜像逐字节提取，匹配即等于「该实例前端就是官方某版产物」。
③
implicit-range 区间
资产指纹命中多个版本（6 组相邻版本前端 bundle 相同、不可区分）时，返回候选区间而非单值，如 2026.3.7 | 2026.3.8。诚实表达不确定性。

为什么这套版本机制是可信的：真实公网验证 + 独立裁决

用第三方测绘平台 ClawSec 列出的境内实例与本地 FOFA 数据交叉补全后实测：严格可一对一比对的样本中版本判定全部正确。对双方分歧的少数实例，不以自有指纹库自证，而以官方镜像作独立裁判——目标实测资产名逐字节比对官方各版镜像产物，确认分歧均为对方平台版本字段陈旧。这条「官方镜像独立裁决」是版本结论可被第三方复现、不循环论证的关键。

04

命令行交付形态：对标 nmap 的 CLI、日志与报告

它终归是个扫描器，应当像 nmap 那样作为命令行工具交付：灵活的目标输入、可控的分级日志、人读与机读两种输出、扫描汇总报告。下面给出 CLI 形态，以及对单个指定 IP，各种真实情况下日志实际长什么样——日志样例的字段、版本号、资产名均取自真实公网/容器探测结果，非虚构。

目标输入：四种形式

形式
示例

1.2.3.4

targets.txt

每个 * 枚举该位 0–255；两个 * 即 256×256 = 65,536 个

展开后的目标总数不设上限——用户对自己输入的范围负责（通配/CIDR 一不小心可达千万级，由使用者把控）。

CLI 参数与配置

# 目标可为：精确IP / 文件 / 通配IP / CIDR（自动识别）
ocprobe 203.0.113.0/24 --port 18789,80,443 --concurrency 16

# 端口      --port 18789,80,443        # 支持多端口
# 并发      --concurrency 16           # 全局同时探测数
# 单IP超时  --timeout 8s               # 对一台主机探测的最长等待
# 任务超时  --deadline 30m             # 整个扫描任务的总时长上限，到点即停、已扫出结果
# 单IP限速  --rate 50                  # 对单个目标IP的每秒发包上限（可选，礼貌/兜底）
# 详细日志  -l                         # 终端精确打印每个IP的完整研判依据（类似下文情况A–F）
# 续扫      --resume                   # 断点续扫
# 报告      --report out.html          # 从数据库生成报告，按文件名后缀定格式（.html/.json/.csv/.txt）

两个超时是不同维度，互不替代：--timeout 管「单台主机别死等」，--deadline 管「整个任务别跑太久」（扫大网段时尤其有用——到点收工、已探的照常出结果）。--rate 是单个 IP 的每秒发包上限，不是全局——单台一轮探测才 5 个请求（1 WS 升级 + 4 GET），50/秒 是远低于会触发目标防御的宽松兜底，主要防异常重试风暴；真正面向大网段的总量控制由 --concurrency 承担。

默认终端输出 vs -l 详细输出

不带任何参数，扫描时终端也会逐目标打一行结果——这是默认行为，不是可选项。每行说明该目标的结局：不可达 / 端口关闭 / 超时 / 非 OpenClaw（False）；或确认是 OpenClaw（True）并给出判定版本。加 -l 则对每个 IP 精确打印完整研判依据（命中了哪些测试项、各属什么证据级别、如何依白名单得出结论），详细度类似下文情况 A–F。

# 默认输出（无参数）——每个目标一行
106.53.198.94:18789   OpenClaw   version=2026.3.7
120.226.246.96:18789  OpenClaw   version=2026.3.1
106.52.205.99:18789   not-openclaw   端口关闭（connection_refused）
198.51.100.7:18789    not-openclaw   仅命中 favicon+title（弱证据，不足以判定）
203.0.113.9:18789     unreachable   超时
# 所有默认值集中存放在扫描器代码的 config.toml；命令行参数覆盖之
[probe]
rate            = 50        # 单 IP 每秒发包上限
timeout_per_ip  = "8s"      # 单 IP 探测超时
concurrency     = 16        # 全局并发

[scan]
deadline        = "0"       # 任务总超时，0 = 不限
ports           = [18789]   # 默认端口

[output]
verbose         = 0         # 默认日志详度（0/1/2 对应 默认/-v/-vv）

研判是二元的：True / False，没有「可能」

扫描器给的是确定结论——是 OpenClaw（True）或不是（False），不输出「可能是」这种骑墙态。证据分级（确证/强/弱）仍然保留，但它的作用是决定落到 True 还是 False，而不是作为一档中间结论。七个测试项编号如下，True 由一个明确的「证据组合白名单」判定：

编号
测试项
证据级别
T1
control-ui-config 200 + serverVersion
确证级
T2
WS connect.challenge（WS 协议面）
强证据
T3
control-ui-config 401（HTTP 路由面）
强证据
T4
/healthz 特征体（HTTP 路由面）
强证据
T5
favicon MD5
弱证据
T6
首页 title
弱证据
T7
响应头三件套
弱证据
// True 当且仅当满足以下任一（其余一切组合 = False）
C1:  T1                         # 确证级单独成立：服务端运行时自报版本，伪造=真实现，无意义
C2:  T2 && ( T3 || T4 )        # WS 强证据 + HTTP 路由面强证据，跨表面双强
  • 为什么纯弱证据一律 False——T5/T6/T7 都是静态文件或常见头，蜜罐全抄齐也没有任何运行时/协议行为。这正是钓鱼站/镜像站的特征，判 True 即被骗。它们只在详细依据里列出，不计入达标。
  • 为什么单个强证据不够——只有 T2（WS）或只有 T3（401）都可能被针对性伪造一个。要求跨两个不同协议表面的双强证据，伪造者得同时仿 WS 子系统与 HTTP 端点子系统，代价高到不如装个真 OpenClaw。
  • 为什么 T3∧T4 不判 True——两者都在 HTTP 路由表面，不跨表面，可能是只仿了 HTTP 端点、连 WS 都没有的残缺仿制品。它落 False，但详细依据会标「命中两个 HTTP 强证据、WS 未响应、疑似」，供人工复查。
  • 这是偏保守（高精确、低召回）的设计——为协同披露普查服务，宁可漏掉「只单面暴露」的真实例，也绝不把仿制品判 True。所有 False 目标带完整证据进数据库，可随时复查或调白名单重判。

扫描结果入库：留下完整请求与响应，报告才有据

扫描器本质是「对每个目标跑若干测试项」。每次扫描的结果都写入数据库，每条记录保存：测试时间、测试对象、每一项测试项的完整请求与响应、最终二元研判结果。完整请求/响应是关键——有了它，任何一条研判都能被复查（为什么判 True/False）、争议可追溯、报告有据，也为日后调整白名单后重判留下原始材料。

// 每个目标一条记录，每个测试项一条子记录（完整请求+响应原文）
scan_result {
  ts            "2026-06-16T11:20:03Z"      # 测试时间
  target        "106.53.198.94:18789"      # 测试对象
  verdict       true                         # 最终研判（二元）
  matched       ["T1","T2","T4","T5","T6"]   # 命中的测试项
  rule          "C1"                         # 触发的白名单条件
  version       "2026.3.7"   version_source "direct"
  probes [
    { id "T2"  request "<WS 升级请求原文>"  response "<首帧原文>"  hit true }
    { id "T1"  request "GET /__openclaw/control-ui-config.json"
            response "<状态码 + 响应头 + body 原文>"  hit true }
    …… 每个测试项都留完整请求与响应原文
  ]
}

生成报告：--report 按文件名后缀定格式

报告从数据库生成，不在扫描时即时拼装——扫描只管落库，出报告是独立动作（可对历史扫描随时出）。格式由 --report 给的文件名后缀决定：

ocprobe --report out.html     # 人读报告（汇总 + 每目标研判依据）
ocprobe --report out.json     # 机读，含完整请求/响应，供二次处理
ocprobe --report out.csv      # 表格：target, verdict, version, matched, ts
ocprobe --report out.txt      # 纯文本摘要

报告话术：测试项 × 状态 → 模板渲染成分析段落

数据库里存的是原始请求/响应，太硬，不能直接给人看。报告生成时走一层模板渲染：每个测试项穷举它「有语义区别」的所有状态，每种状态配一段半写死的话术（留变量槽由实际值填入）；渲染时把目标各测试项的实际状态逐个匹配到话术、填值，拼成一段连贯的分析。话术与判定逻辑解耦，单独放 report_templates.toml（与 config.toml 平级），非技术人员也能改措辞、不必动代码或重编译。下面的状态枚举均取自真实探测中实际出现过的状态，非设想。

06

优化方案：分梯队，每条对着一个实测失效

按「实测证据强度 × 成本 × 收益」排序。可信级别标注：实测证实 已由真实数据证明问题存在、改法明确；需先验证 方向合理但效果待真实样本确认。改动集中在 prober，不依赖任何不可行前提。

① 失败原因分类 实测证实 第一梯队

当前探不到目标时只记一个空结论，无法区分真下线、超时、连接拒绝、不可达（见 §04 情况 F）。

result.go 加 error_type（timeout / connection_refused / unreachable / down）；probe.go 在连接失败时按错误类型填写。

→ 解锁「真下线 vs 假下线」诊断，是下面 ② ③ 两项的前提。改动约 20 行，零风险。

② 品牌变体识别 实测证实 第一梯队

实测出现 title 为 Clawdbot Control（改名前旧名）、Moltbot Control（分叉）的实例，确认是 OpenClaw 却取不到版本。

判定与指纹库加入品牌变体的 title 与资产映射；版本反推对变体品牌的资产名同样查库。

→ 直接修复实测中「is_openclaw=true 但 version 为空」的落空。

③ 隐位段采样验证替代全枚举 实测证实 第一梯队

测绘源隐藏一位地址，补全一对多（67.9%）。当前对下线候选要在整段枚举 256 个，成本高。

基于实测规律「同隐位段内活实例版本一致性 100%」，改为采样验证：先探该段 2–3 个候选，确认版本一致即代表全段，不必逐个全扫；补全选错对象时也在段内有限枚举确认而非直接判下线。

→ 砍掉约一半冗余探测；同时修复「84.6% 假下线」。这是大规模真实测试才提炼出的、纯数据驱动的优化。

④ 资产反推加相似度兜底 需先验证 第二梯队

资产指纹当前要求集合精确相等，差一个文件名（CDN 改名、漏采一个资产）就整体落空、退回无版本。

精确匹配失败后，按观测资产集与库中各版的 Jaccard 相似度，≥0.75 时返回候选并标 implicit-fuzzy。落地前须先在 71 版库上统计 js/css 单独命中的碰撞率，确认不会引入误配。

→ 缓解部分升级/CDN 改名导致的整体丢失，提升反推召回。

⑤ 探测并行化 + 可配超时重试 实测证实 第二梯队

probe.go 当前串行跑 HTTP 与 WS 探测（最坏 16 秒），固定 8 秒超时对不稳定公网偏短、无重试——直接抬高「假超时下线」率与拉低吞吐。

HTTP 与 WS 探测并行；超时与重试次数参数化（如 ConnectTimeout、Retries、RetryDelay）。

→ 提升真实公网扫描吞吐，降低把慢目标误判成下线的概率。

⑥ 留存响应头与完整 config 键集合 需先验证 第二梯队

http.go 读到完整响应头却几乎全丢（Server、ETag 等）；control-ui-config 只取 serverVersion 一个字段，完整 JSON 键集合未看。

Evidence 增 map[string]string 存关键响应头快照；200 时多存 config 的 JSON 键集合。先在多台同版本实例上验证这些头/键的来源与稳定性（区分 OpenClaw 自身 vs 反代注入），再决定是否纳入判定。

→ 低成本留证据；可能为版本判别提供补充维度。顺带清理 consts.go 中定义却未用的 detectConfidenceFloor 死常量。
