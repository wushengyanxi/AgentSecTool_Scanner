AgentSecTool Scanner · 项目文档

OpenClaw 公网暴露扫描器：系统结构、判定与版本测定机制、数据记录方式

本文档说明的内容

本项目是一个针对 OpenClaw 网关公网暴露面的扫描系统：在授权范围内发现暴露的实例、只读地判定其是否为 OpenClaw、测定其版本，并将结果归档，服务于后续的漏洞协同披露。本文档面向接手或评审本项目的工程读者，逐层说明系统结构、探测判定机制、版本测定机制、目标来源、数据记录方式与运行方式，并对每项能力标注其实现位置与当前状态。

两个贯穿全文的口径需在阅读前明确：其一，扫描器的正式能力只有“发现后的只读判定与版本测定”，由 prober/ 承担；FOFA、ClawSec 与 scan_test/ 是被动测绘数据源与验证工具，为扫描器提供探测目标、构建测试用例集，并非扫描器的判定功能本身。其二，项目内有四种数据写入方式（候选目录当前态、每日快照、扫描台账当前态、观测明细追加），分别服务于不同用途，第 06 节逐一说明各自的做法与设计取舍。

01

系统结构：每个模块的职责

系统按职责分为若干 Python 模块与一个 Go 探测引擎，各自独立、通过文件与本地 SQLite 库衔接。下表的目录树标注每个模块的角色；其中 prober/ 是扫描器本体，其余模块分别承担发现、测绘供料、落地归档、可视化与运维。

02

架构与数据流：正式主链路与供料旁路

系统存在两条数据链路。正式主链路由发现、探测、落地构成，是扫描器的产品路径；供料与验证旁路由 FOFA、ClawSec 与 scan_test/ 构成，向探测引擎提供目标或验证用例，但不参与判定。两条链路最终都汇入同一个探测引擎 ocprobe——判定与版本测定只发生在这一处。

实线＝正式主链路；虚线＝可选的供料旁路。FOFA 与 ClawSec 各自把外部平台候选归档到独立库，需要时导出为目标清单交给同一个 ocprobe 验证；scan_test/ 的枚举验证同理。无论目标来自哪条链路，判定与版本测定只在 ocprobe 内发生，反白方框即扫描器本体。

关于被动种子的两套实现

正式主链路中的被动种子由 discovery/passive.py 的实时 FOFA / Shodan 查询提供，直接产出 candidates.csv。它与 fofa/、clawsec/ 模块是两套独立实现：后者面向归档、按省资产规划与平台-自测版本双源对比，是供料与分析侧的能力，不读取也不依赖 discovery 的查询路径。

03

判定机制：如何确认一台是 OpenClaw

判定不依赖任何单条特征，而是要求来自多个相互独立的协议表面的证据联合成立。OpenClaw 网关在三个表面上留有可被动观测的痕迹，探测器分别采集，再按一个明确的证据组合白名单做二元研判。

三个观测表面与对应的测试项（编号 T1–T7，定义见 prober/openclaw/result.go）：

  • WebSocket 协议面：连接 ws://host:18789/ 后，服务端立即推送一帧 connect.challenge 事件（T2，强证据）。此行为在其他服务上罕见，是判别力最高的单条信号。
  • HTTP 路由与头面：/healthz 返回特征体 {"ok":true,"status":"live"}（T4，强证据）；/__openclaw/control-ui-config.json 在已防护实例上返回 401，表明端点存在（T3，强证据）；若该端点返回 200 并携带 serverVersion，则为确证级（T1，服务端运行时自报）。
  • HTTP 静态资源面：首页 <title> 标记（T6）、favicon.ico 的 MD5（T5）、响应头三件套（T7）。三者均为弱证据——可被静态仿冒，故不单独参与达标，仅在报告中列出。

判定规则：证据组合白名单

研判是二元的（是 / 否），不输出概率分值。当且仅当满足以下任一条件时判为 OpenClaw，其余一切组合均为否（见 prober/openclaw/signals.go · evaluate()）：

达标条件
C1：T1 —— 确证级单独成立（control-ui-config 自报版本，仿冒成本等同于真实实现）
C2：T2 && (T3 || T4) —— WebSocket 强证据 叠加 HTTP 路由面强证据，跨表面双强

规则的核心是跨表面：同一表面内的多条信号容易被一并构造仿冒，唯有来自不同协议表面的强证据同时成立，才具备抗仿冒能力。纯弱证据（T5/T6/T7）无论命中多少都不参与达标，仅供报告说明。

探测全程只读

探测只发送 HTTP 升级请求与无鉴权 GET，绝不发送 connect 帧或 config.apply——后者本身属于漏洞利用面。这一约束由单元测试固化：probe_test.go 断言探测过程发送的字节中不含越界帧。

探测流程：先过可达性门槛

对单台目标，Probe() 先做一次 TCP 连接作为可达性门槛：连不上的目标（被墙、黑洞、端口关闭）直接返回失败原因，不再发 HTTP/WS。这一步是必要的——公网扫描里不可达目标占绝大多数，若对它们也走完整流程，每个都要白白耗掉四个 HTTP GET 加 WebSocket 各自的超时（串行累加可达数十秒），把固定数量的并发 worker 长期钉死、有效并发塌方。可达的目标才继续走 HTTP 与 WebSocket 探测、再做研判。HTTP 探测复用一个全进程共享的连接池（而非每台新建），使空闲连接受上限与超时约束、不会在高并发下累积成耗尽文件描述符的孤儿连接。

实现位置
openclaw/probe.go · Probe() —— 编排：可达性门槛 → HTTP/WS → 研判，产出 Result
openclaw/probe.go · dialReachable() —— TCP 可达性门槛，区分 timeout/refused/unreachable/down
openclaw/ws.go · probeWS() —— WebSocket 升级，读取首帧，只读
openclaw/http.go · probeHTTP() —— 四个 GET（复用进程级共享连接池），抽取标题 / favicon / healthz / CSP / 响应头
openclaw/signals.go · evaluate() —— 证据组合白名单二元研判

已验证 对本地靶机与真实公网实例，C1 / C2 判定与单元测试一致。完整的判别力与证据分级论证见 docs/扫描器优化与研判机制.html。

04

版本测定机制：直读与隐式指纹反推

版本既不在页面标题中，也不在 favicon 中——favicon 自某一早期版本加入后跨版本恒定，只能用于判定、不能区分版本。版本测定按优先级走两条路径，均为只读观测。

  • 直读（高置信）：GET /__openclaw/control-ui-config.json 若返回 200，响应体中的 serverVersion 字段即为精确版本。仅未启用鉴权（auth=none）的实例可得；已防护实例此端点返回 401。
  • 隐式反推（中置信）：当端点返回 401 时，改用主机无需鉴权即可观测、且随版本变化的量去匹配指纹库。主信号是首页引用的 assets/index-<hash>.js/.css（构建产物的内容哈希，同版本恒定、跨版本不同），备用信号是 CSP 的 sha256。匹配时以资产哈希集合的精确匹配优先。

指纹库的来源

指纹库由 fingerprint/build_corpus.py 离线构建：在容器内逐个启动已知版本的 OpenClaw，用 ocprobe 采集其资产哈希、CSP、favicon，写入 fingerprints/fingerprints.json。指纹库本身按版本号去重，重新采集同一版本时整条覆盖。

实现位置
openclaw/version.go · determineVersion() —— 先尝试直读，否则走隐式反推
openclaw/fingerprints.go · MatchImplicit() / LoadFingerprintDB() —— 资产哈希 / CSP 匹配版本
fingerprint/build_corpus.py · record_signature() —— 逐版本采集签名入库
fingerprints/fingerprints.json —— 版本指纹库（version → 资产哈希 / CSP / favicon）

部分 机制完整、单元测试通过、真实实例可直读出版本号。当前指纹库收录版本有限，对未收录版本的已防护实例，隐式反推可能匹配不到、版本记为未知；扩充指纹库即可覆盖更多版本。

05

目标来源与测试用例集

扫描器需要被探测的目标清单。目标有两个来源层次：正式流水线自带的发现层，以及若干被动测绘数据源。后者是本节要厘清的重点——它们为扫描器提供探测目标、构建测试用例集，本身不发探测包、不做判定，不属于扫描器的正式判定能力。

正式发现层：discovery

discovery/ 是正式流水线的发现层，回答“哪些 IP:端口值得探测”，产出 candidates.csv。它并取两路：主动扫描（masscan / ZMap 无状态 SYN 扫描，或内置的 async connect 扫描）与实时被动种子（passive.py 直接查询 FOFA / Shodan 接口）。两路结果经 merge.py 合并去重、剔除保留网段与黑名单后写出。

被动测绘数据源：FOFA 与 ClawSec

定位

fofa/ 与 clawsec/ 是两个独立的被动测绘数据源模块，各自维护独立的库。它们的职责有二：其一，把外部平台返回的候选原样归档，需要时导出为 candidates.csv 作为探测器输入；其二，做按省资产规划与“平台标注版本 vs 自测版本”的双源对比。它们本身不发探测包、不做 OpenClaw 判定——确认与版本测定一律由 prober/ 完成。

具体而言，FOFA 与 ClawSec 仅提供平台已索引的候选与平台自己的标注；这些数据等于“哪里可能有 OpenClaw”，不等于“确认是 OpenClaw”，也不含本项目认可的版本。候选是否成立、版本是多少，仍由探测器逐台只读探测得出。这两个模块的产物有两种去向：一是导出为目标清单（可选旁路，喂给 ocprobe），二是留作纯分析（如跨库重叠统计、按省份的版本分布），后者根本不进入扫描判定数据流。

实战验证目录：scan_test

scan_test/ 是与扫描器本体分离的实战验证目录。其脚本拉取平台公开列表作为目标清单（脚本注释明确“仅打公开 API，不探测任何 IP”），再对隐位 IP 段调用真实的 ocprobe 做只读枚举，并与平台标注版本双源核对。它用扫描器跑验证用例，扫描能力本身仍在 ocprobe。该目录的产物含真实 IP，全部纳入 .gitignore，不进入产品库。

一句话边界

prober/ 是扫描器本体，承担发现之后的只读判定与版本测定。fofa/、clawsec/、scan_test/ 不参与判定：它们负责被动测绘归档、构建探测目标与测试用例集，以及双源对比与资产规划——是扫描器的供料与验证侧能力，而非核心功能。

实现位置
discovery/active.py · passive.py · merge.py —— 主动 / 被动发现与合并去重
fofa/pull.py · pull() / export_candidates() / province_versions() —— 归档、导出、按省版本分析
clawsec/pull.py · pull() / overlap_with_fofa() —— 每日快照归档、跨库重叠

06

数据记录如何写入：四种方式与各自的设计取舍

项目里有四种相互独立的数据写入方式，它们落在不同的库表中，对“同一条记录再次被处理时如何变化”也给出不同的答案。这种差异不是疏忽，而是因为每种数据服务于不同的目的——候选目录要的是最新一份、暴露面观察要的是逐日历史、扫描台账要的是当前态与不可变证据兼得。本节逐一说明每种写入方式做什么、怎么做、以及为什么这样设计，最后归纳它们共同遵循的原则。

FOFA 候选目录：只保留最新的一份

FOFA 候选写入 fofa_candidates 表，主键是 (ip, port)，同一台主机的同一端口在表中永远只占一行。再次从 FOFA 拉到同一条目时，last_seen、归属省市、原始响应等字段就地刷新为本次的值，只有 first_seen 保留首次拉取的日期。

这样设计是因为 FOFA 候选的角色是一份“当前可探测目标的名录”——它要回答的是“现在 FOFA 索引里有哪些疑似 OpenClaw 的主机”，而非“这台主机历史上每次被 FOFA 看到时是什么样”。把它做成单份当前态，目录始终保持最新、体积可控，导出为探测目标时也不必在多份历史里挑选。代价是旧的原始响应被覆盖后不可追溯；由于这份数据本就只用于产生探测目标、真正的证据在扫描台账里另有保存，这个代价是可接受的。

ClawSec 每日快照：每天留一份完整镜像

ClawSec 数据写入 clawsec_snapshots 表，主键是 (snapshot_date, id)——在实例标识 id 之前多了一个快照日期 snapshot_date（取自平台自报的 lastScanTime）。这意味着同一个实例在不同日期各占一行，旧日期的行原样保留；只有同一天内同一实例被重复拉到时，才会更新版本、运行状态、漏洞计数等少数随时间变化的字段，其余字段保留当天首次入库的值。

这是全项目唯一按天分版、保留历史镜像的数据。之所以为 ClawSec 单独采用这种方式，是因为它的用途正是“观察暴露面随时间的演变”：哪些实例长期在线、版本如何随日期推移、某天新增或下线了多少——这些问题只有保留逐日的完整快照才能回答。如果像 FOFA 那样只留当前态，跨日分析就无从谈起。断点续传按 (snapshot_date, page) 记录已完成的页，中断重跑会接着拉；某一天的全量已入库后再次运行会幂等跳过，因此正常情况下几乎不会触发对已有行的覆盖。

扫描资产台账：当前态，但保护历史信号

扫描结果写入 store 的 assets 表，资产身份是 IP 与端口的组合（asset_id = sha1("ipport:{ip}:{port}")），每台暴露主机一行。当同一台主机被复扫、结果再次入库时，assets 做的是合并式更新而非整行替换：last_seen 刷新为本次时间，观测计数加一，is_openclaw 取新旧两值中较大的，版本与地理字段在解析到新值时才覆盖、否则保留旧值，研判档只在更强时升级而不会被更弱的复扫降级，first_seen 始终保持首次入库的日期。

这种“当前态 + 历史信号保护”的混合策略，对应实例收集的实际需要。assets 要回答的是“截至目前我们掌握了多少台 OpenClaw、它们当前的版本与归属”，所以主体是当前态、每台一行便于统计。但若简单地整行覆盖，一次失败或受限的复扫就会把先前更强的研判与已取到的版本抹掉，台账会越扫越差。因此对“只会越来越确定”的几类信号——首见时间、最强研判档、是否曾确认为 OpenClaw——采取只增不减、取最强档的规则加以保护。需要注意由此带来的一个口径：当前态保留的是历史上最强的研判，并不保证此刻一定在线；要判断“此刻是否在线”，应查下面的逐次观测记录。

观测与探测明细：只追加，永不修改

每次探测除了更新 assets，还会向 observations 追加一行带时间戳的观测，并向 probe_records 追加该次每个测试项的完整请求与响应原文。这两张表用自增主键、不做去重，只追加、从不更新或删除——同一台主机被探测多少次，就留下多少行。

它们是整个系统的不可变证据底账。研判结论可能随版本演进而调整，但“某时刻对某主机发了什么、收到了什么”是客观事实，必须原样留存以供复查与争议追溯；同时它也是时间维度分析的数据基础——按省份、按版本随时间的分布报表都依赖这条逐次的观测流水。正因为如此，这两张表绝不覆盖：覆盖会销毁证据。这里的“逐次”也要与 ClawSec 的“逐日”区分——observations 是每探测一次追加一条，而非每天一份整库镜像。

两个常见疑问的直接回答

是否每天分别有一版镜像？只有 ClawSec 是按天分版的每日镜像；FOFA 候选与扫描 assets 都是每个身份键只有一行的当前态；observations 保留历史，但它是逐次追加的观测流水，不是每日整库镜像。

曾处理过的 IP 再次被处理，记录会不会被覆盖？看落在哪张表：FOFA 候选会就地覆盖（仅首见时间保留）；ClawSec 跨日各留一行、互不覆盖（仅同日同实例覆盖少数易变字段）；扫描 assets 合并式更新，首见时间、最强研判档等历史信号受保护；observations 与 probe_records 永不覆盖，每次都新增一行。

共同遵循的原则

四种方式虽各有取舍，但共享几条一致的约定，理解它们有助于判断新数据应当如何落库。其一，身份键以 IP（或 IP 加端口）为核心，扫描资产不按页面内容指纹跨 IP 合并，以免把不同主机当成同一台。其二，first_seen 这类“首见”时间一经写入便不再改动，last_seen 一类“最近见到”的时间则每次刷新。其三，凡保留原始响应的字段都是可追溯证据，覆盖须谨慎——这也是 FOFA 候选只留最新原始响应、不可追溯历史时需要明确标注的原因。其四，地理列遵循“解析到新值才覆盖、解析不到则保留旧值”，回填时不以空值抹掉已有的城市。临时产物（candidates.csv、探测器输出的 results.jsonl、scan_test/ 的 JSON 文件）不属于权威数据，它们或被整文件重写、或仅在内存中去重，须经入库后方成为权威记录。

地理富化：IP 到城市

扫描到的目标在入库时按 IP 解析到物理城市，写入 assets 表的 country / region / city 列，供看板的地理分布展示与按城市的资产规划。

来源是 MaxMind GeoLite2-City 离线库——城市级地理定位的通用基准，离线、可复现。解析在 store 入库时随写完成；另有回填脚本用于给存量资产补全或在库更新后重刷。城市归属是离线库依据 IP 推断的近似值，城市级存在误差（云与 IDC 网段尤其明显），非权威、非长期固定，仅作富化展示，不作为判定依据。该模块是软依赖：离线库或 geoip2 缺失时优雅降级、地理列留空，不阻断入库。

地名口径：只取英文，本地映射成简体中文

GeoLite2 自带的中文地名不可信：部分城市没有中文条目而回退英文，部分中文名不规范、甚至把某些 IP 标到生僻或错误的地名。直接采用它的中文名，页面就会混入英文、半中半英、乃至错误的地名，命名口径不受控。

因此 city 与 region 只取库的标准英文名，再经一张本地映射表转成简体中文——命名口径完全由本项目掌握。映射表是 geoip/cn_names.py 里的两个字典（英文→简体中文），入库时查表：命中则写中文，未命中则原样写入英文原文，既不丢数据也不报错；页面上出现英文，即表示该名尚未收录，补进映射表再重刷即可。country 仍取库的中文名——国家级简体名可靠，无需自建映射。

映射表用 Python 字典而非 JSON 或 CSV，是为了入库热路径上的性能：模块顶层 import 一次即把两个字典常驻内存，对每个 IP 的查表是 dict 的 O(1) 操作，无文件解析开销。

实现位置
geoip/lookup.py · GeoResolver.lookup() —— mmdb 查询封装，取英文名后查映射表，软依赖可降级
geoip/cn_names.py · CITY_ZH / REGION_ZH —— 英文→简体中文映射表（dict，常驻内存）
geoip/backfill.py —— 存量回填 / 映射表更新或库更新后重刷
store/load.py —— 入库时按 IP 解析并写入地理列（COALESCE 保留旧值）

看板：只读数据展示端

看板是系统的数据消费端，把入库的扫描结果呈现为可浏览的页面：实例总数与版本分布、按可信级别的分桶、地理分布地图、资产列表，以及单台主机的完整探测报告。它只读查库、不参与扫描与判定，默认只监听本机回环、不对公网暴露。

后端：纯标准库的只读查询服务

后端是一个基于 http.server.ThreadingHTTPServer 的小服务，零第三方依赖（仅用 sqlite3 / json / hashlib 等标准库），默认监听 127.0.0.1:8787、只读查 data/scanner/scan_results.sqlite，全程没有任何写库操作。它对前端暴露几个查询接口，并伺服前端用到的静态资源。

资产列表里双击一行，会在其下展开该主机的探测报告（调 /api/report）：顶部是研判结论、版本与命中标记（T1/C1 等带悬停含义），其下逐项列出探针——每项可再展开看观测描述、原始请求与响应，三处用统一的特征词高亮相互对应；末尾是研判小结，把达标逻辑（C1/C2、证据级别）收口讲一次。一次只展开一台，列表更清爽。

python3 -m progress：主动扫描进度

本文档的口径以代码为准：扫描器的正式能力是 prober/ 的只读判定与版本测定；FOFA、ClawSec 与 scan_test/ 是供料与验证侧的测绘数据源，不参与判定。数据写入则按用途各取所宜——候选目录只留最新一份、暴露面观察按日保留镜像、扫描台账维持当前态、观测明细只追加不改，各自的取舍见第 06 节。各项能力的实现位置见每节的“实现位置”。
