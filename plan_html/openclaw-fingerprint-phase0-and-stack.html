OpenClaw 指纹实测结论，以及测绘平台、技术栈与暴露面板数据模型的选定

结论先行 · TL;DR

在容器内以 --allow-unconfigured 跑起 OpenClaw 2026.5.17（仓库 HEAD），对其外部可观测面做了只读探测。判定（是不是 OpenClaw）已可在零鉴权下做到极高可靠：WebSocket 一连上，服务端立即推送 connect.challenge 事件；/healthz 回 {"ok":true,"status":"live"} 且带一组特征响应头；首页 <title>OpenClaw Control</title> 与 favicon.ico 的 md5 固定。

版本判定分两种处境：疏于防护的实例（auth=none）在 /__openclaw/control-ui-config.json 直接吐出 serverVersion；已设令牌/配对的实例该接口回 401，握手里的版本也拿不到——此时只能走隐式指纹（首页引用的 assets/index-<hash> 资产哈希、CSP 允许域、响应头集合、协议版本）反推版本区间。两条路都按 CalVer 映射到 CVE。本文给出实测信号清单、测绘平台推荐、选定的技术栈与存储，以及对标范例那张暴露面板的数据模型。

01

判定逻辑：单条信号的判别力、获取难度与组合置信度

判定一台主机是不是 OpenClaw，不靠单条信号，而靠多条联合。要把它说清楚，得回答三件事：每条信号单独能带来多大置信、在真实公网扫描里多容易拿到、以及不同组合能堆到多高。下面先逐条给出，再给组合阈值与最小"≈100%"集。文中置信度均为工程先验估计，最终以 Phase 0 真值语料的精确率/召回校准。

获取难度口径——易：一次匿名 HTTP GET，反代友好，多数已被 Censys / Shodan / FOFA 被动收录，无需自己发包即可命中；中：需主动完成 WebSocket 升级（必要时先 TLS），反代不转发 WS 时取不到，被动平台一般没有。

辨析 · 单个 CSP 域不是指纹

CSP 里出现 api.openai.com 并不能证明是 OpenClaw——任何调用 OpenAI 的应用都有它；tweakcn.com 单独也几乎没有判别力。这一行真正的判别力来自两点：一是对 /__openclaw/control-ui-config.json 这条 OpenClaw 专有路径返回 401（说明服务端"认识"该路由），二是整串 CSP 的逐字节匹配。原则推广到所有"内容可被复制"的信号：看专有结构与整体，不看任何可复用的零件。

A · WS connect.challenge单独 ≈0.99

连上 ws://h:18789/ 即收到未经请求的 {"type":"event","event":"connect.challenge","payload":{"nonce":…,"ts":…}}。

判别力来源：开局未经请求即下发的 bespoke 挑战帧，字面量加 JSON 结构近乎唯一，正常服务与克隆静态页都不会有。单条判别力最高；残余不确定仅来自蓄意蜜罐可仿。

先验与护栏 · 不是证据

端口 18789 开放：只抬高"值得探测"的优先级；任何东西都能监听它，且常被反代迁走，单独 P(OpenClaw) < 0.05，不能作判定。

路由回落 200：未知顶层路径回 200（SPA 外壳）、而 /api → 404，这是防误报的护栏——判定看具体响应内容，绝不把猜测路径上的裸 200 当作"端点存在"。

独立性 · 为什么要跨表面组合

置信度相乘只在不同协议表面的信号之间近似成立。A 与 F 都出自 WebSocket，伪造一个就伪造另一个，叠加几乎不增信；B 与 C 都来自同一个 SPA 外壳，也高度相关。真正把置信推向 1 的，是跨表面组合：WebSocket 协议面（A）× HTTP 静态资源面（B / C）× HTTP 路由与头行为面（D / E）。克隆一张静态页很容易，要同时正确实现 OpenClaw 的 WS 握手协议则很难——这就是跨表面的价值。

02

探测与判定：只读、多信号、低误报

把 01 的信号组织成一次判定，遵循三条原则——只读良性、多信号交叉、不被裸状态码欺骗。

只读、良性。探测止于"开局握手 + 几个无鉴权 GET"。WebSocket 只读到 connect.challenge 与未授权拒绝信封即可识别，绝不发送任何会改状态或试图越权的帧——尤其绝不触碰 config.apply（其本身即 CVE-2026-25593 的利用面）。这条在 ZGrab2 模块里要写成硬不变量。

多信号交叉。任一信号都可能被蜜罐或巧合命中，因此判定要求至少两个独立信号吻合：例如 connect.challenge ＋ /healthz 体，或 /healthz 体 ＋ 首页标题/favicon。connect.challenge 单独已足够强，但仍以双证据入账以压低误报。

不被裸状态码骗。由于未知路径回落到 SPA 的 200，判定逻辑必须核对响应体特征（{"ok":true,"status":"live"}、OpenClaw Control、favicon 哈希、那串 CSP），而不是"某路径返回 200 即认定为某端点"。

面对反代/CDN 时，纯 WebSocket 协议探测可能打不穿，但 HTTP 侧的标题、favicon、CSP、响应头往往随响应透传——因此 HTTP 指纹与 WS 指纹互为补充，缺一不可。

03

版本判定的两种处境

"检查每一台的版本"在公网上不是一条路走到底。实测表明，能不能直接读到版本，取决于该实例的防护姿态。下图是判定流程，随后两张卡片分别说明两条路。

方框为步骤，箭头为流向，灰底框是分支判断，深色终端框是出口（版本落到 CVE 映射并入库）。左路对应疏于防护的实例，可直接读到精确版本；右路对应已防护的实例，只能靠随版本变化的可观测量反推区间。两路最终都带置信标签汇入同一套记录——面板上"存在漏洞的暴露实例"正由此而来。

左路 · 疏于防护（auth=none）

/__openclaw/control-ui-config.json 返回 200，体含 serverVersion（源码 resolveRuntimeServiceVersion 注入，即产品版本如 2026.5.17）。一次无鉴权 GET 即得精确版本，高置信。

这正是 CVE-2026-25253 那批"裸奔"实例所属的姿态，公网占比不低，是版本数据最干净的来源。

右路 · 已防护（令牌 / 设备配对）

该接口 401，hello-ok（含 server.version、protocol: 4）也需配对才拿得到 → 无直接版本串。改用隐式指纹：首页引用的 assets/index-<hash>.js/.css（按构建内容哈希、随版本变化）、CSP 允许域、响应头集合、协议版本号。

这些信号→版本的对照表，需用指纹库承载：在容器里逐个跑起各发布版本、记录其外部签名（正是本次所用方法），产出"签名 → 版本/版本区间 + 置信度"。协议版本 4 较粗，只作下界/区间。

07

复现与下一步

这次实测的最小复现如下（注意 compose 默认 command 缺 --allow-unconfigured 会因 "Missing config" 崩溃重启，故直接用 docker run 起最小可指纹形态）。同一套命令就是逐版本建指纹库的真值 harness。

# 1) 构建镜像（首次，产出 openclaw:local）
docker compose -f docker-compose.yml --project-directory . build

# 2) 以最小可指纹形态起一个实例（已设令牌 = 已防护姿态）
docker run -d --name oc-fp -p 18789:18789 \
  -e HOME=/home/node -e OPENCLAW_HOME=/home/node \
  -e OPENCLAW_GATEWAY_TOKEN=$(openssl rand -hex 32) \
  openclaw:local node dist/index.js gateway --bind lan --port 18789 --allow-unconfigured

# 3) 只读探测：判定信号 + 版本信号
curl -s http://127.0.0.1:18789/healthz                              # {"ok":true,"status":"live"}
curl -s -i http://127.0.0.1:18789/__openclaw/control-ui-config.json # 401 / 或 200 含 serverVersion
curl -s http://127.0.0.1:18789/ | grep -oE 'assets/index-[^"]+'     # 版本相关资产哈希
node -e 'const w=new WebSocket("ws://127.0.0.1:18789/");w.onmessage=e=>{console.log(""+e.data);w.close()}' # connect.challenge
